# Jenkins Zero to Hero - Complete Guide

## What is Jenkins?

Jenkins = A robot that automates your work

**Without Jenkins:**
- You manually build code (30 minutes)
- You manually test (20 minutes)
- You manually deploy (15 minutes)
- Total: 65 minutes of boring work

**With Jenkins:**
- Push code to GitHub
- Jenkins does everything automatically
- Total: 3 minutes, you drink coffee ☕

---

## Part 1: Why Use Docker with Jenkins?

### Old Way (Expensive & Slow)

```
Jenkins Master
    ↓
Worker 1 (always running) → $100/month
Worker 2 (always running) → $100/month
Worker 3 (always running) → $100/month
```

**Problems:**
- 3 computers running 24/7 = $300/month
- Night time: No work, still paying
- Weekend: No work, still paying
- Hard to update (login to each computer)

### New Way (Cheap & Fast)

```
Jenkins Master
    ↓
Docker (creates containers when needed)
    ↓
Container appears → Works → Disappears
```

**Benefits:**
- Container created only when needed
- Work done in 3 minutes
- Container deleted
- Cost: Only 3 minutes of compute
- Easy updates: Change one line in code

## Part 2: Installing Jenkins

### Step 1: Create EC2 Computer

1. Go to AWS
2. Launch EC2 (Ubuntu)
3. Name: "Jenkins-Server"

### Step 2: Connect

```bash
ssh -i your-key.pem ubuntu@YOUR-IP
```

### Step 3: Install Java

Jenkins needs Java to run.

```bash
sudo apt update
sudo apt install openjdk-11-jdk -y
java -version
```

### Step 4: Install Jenkins

```bash
curl -fsSL https://pkg.jenkins.io/debian-stable/jenkins.io-2023.key | sudo tee \
  /usr/share/keyrings/jenkins-keyring.asc > /dev/null

echo deb [signed-by=/usr/share/keyrings/jenkins-keyring.asc] \
  https://pkg.jenkins.io/debian-stable binary/ | sudo tee \
  /etc/apt/sources.list.d/jenkins.list > /dev/null

sudo apt update
sudo apt install jenkins -y
sudo systemctl start jenkins
sudo systemctl enable jenkins
```

### Step 5: Open Port 8080

AWS blocks port 8080 by default.

1. Go to EC2 → Security Groups
2. Edit inbound rules
3. Add rule: Port 8080, Source: 0.0.0.0/0
4. Save

### Step 6: Access Jenkins

Browser: `http://YOUR-IP:8080`

Get password:
```bash
sudo cat /var/lib/jenkins/secrets/initialAdminPassword
```

### Step 7: Setup

1. Paste password
2. Install suggested plugins (wait 2 min)
3. Create admin user
4. Save and finish

✅ Jenkins installed!

## Part 3: Installing Docker

### Step 1: Install Docker

```bash
sudo apt install docker.io -y
sudo systemctl start docker
sudo systemctl enable docker
```

### Step 2: Understanding Users

**Important concept:**

Every program runs as a "user":
- **ubuntu** = You
- **jenkins** = Jenkins program (created automatically when Jenkins installed)
- **root** = Administrator

**Problem:** Only root can use Docker by default. Jenkins can't!

**Solution:** Add jenkins to docker group.

### Step 3: Give Permission

```bash
sudo su
usermod -aG docker jenkins
usermod -aG docker ubuntu
systemctl restart docker
exit
```

**What this does:**
- jenkins user joins docker group
- Now jenkins can use Docker

### Step 4: Restart Jenkins

Browser: `http://YOUR-IP:8080/restart`

Or terminal:
```bash
sudo systemctl restart jenkins
```

### Step 5: Test

```bash
sudo su - jenkins
docker run hello-world
exit
```

**Success:** You see "Hello from Docker!"

**Fail:** You see "Permission denied" (repeat Step 3)

### Step 6: Install Docker Plugin

In Jenkins:
1. Manage Jenkins → Manage Plugins
2. Available tab
3. Search: "Docker Pipeline"
4. Install
5. Restart Jenkins

✅ Docker ready!

## Part 4: Your First Pipeline

### Create Pipeline

1. New Item
2. Name: "My-First-Pipeline"
3. Select: Pipeline
4. OK

### Write Code

```groovy
pipeline {
    agent {
        docker {
            image 'node:16-alpine'
        }
    }
    
    stages {
        stage('Test') {
            steps {
                sh 'node --version'
            }
        }
    }
}
```

### Understanding Code

**Line by line:**

```groovy
pipeline {
```
Start pipeline

```groovy
agent {
    docker {
        image 'node:16-alpine'
    }
}
```
Use Docker container with Node.js 16

```groovy
stages {
    stage('Test') {
        steps {
            sh 'node --version'
        }
    }
}
```
One stage called "Test", runs command `node --version`

### Run It

1. Save
2. Build Now
3. Watch it run!

### What Happens

1. Jenkins asks Docker: "Give me Node.js 16 container"
2. Docker creates container
3. Jenkins runs `node --version` inside
4. Output: v16.14.0
5. Jenkins deletes container
6. Done! (5 seconds)

### Verify

```bash
docker ps
```
No containers running! They're deleted after use.

## Part 5: Multi-Stage Pipeline

Real apps need multiple steps.

### Code

```groovy
pipeline {
    agent {
        docker {
            image 'node:16-alpine'
        }
    }
    
    stages {
        stage('Checkout') {
            steps {
                echo 'Getting code...'
            }
        }
        
        stage('Build') {
            steps {
                echo 'Building...'
                sh 'node --version'
            }
        }
        
        stage('Test') {
            steps {
                echo 'Testing...'
                sh 'echo "Tests passed!"'
            }
        }
        
        stage('Deploy') {
            steps {
                echo 'Deploying...'
                sh 'echo "Deployed!"'
            }
        }
    }
}
```

### How It Works

Stages run ONE AT A TIME:
```
Checkout → Build → Test → Deploy
```

If one fails, rest skip.

**Example:**
```
Checkout: ✓
Build: ✓
Test: ✗ FAILED
Deploy: ⊘ SKIPPED (because Test failed)
```

## Part 6: Multi-Agent Pipeline

Different parts need different tools:
- Frontend needs Node.js
- Backend needs Java
- Database needs MySQL

Solution: Use different containers!

### Code

```groovy
pipeline {
    agent none
    
    stages {
        stage('Backend') {
            agent {
                docker { image 'maven:3.8-openjdk-11' }
            }
            steps {
                sh 'mvn --version'
            }
        }
        
        stage('Frontend') {
            agent {
                docker { image 'node:16-alpine' }
            }
            steps {
                sh 'node --version'
            }
        }
    }
}
```

### What Happens

1. Backend stage: Maven container created → work done → deleted
2. Frontend stage: Node container created → work done → deleted

Each stage uses its own container!

## Part 7: Using GitHub Code

Real projects: Code is in GitHub, not Jenkins.

### Setup

1. New Item → "GitHub-Pipeline"
2. Pipeline script from SCM
3. SCM: Git
4. Repository URL: `https://github.com/YOUR-USERNAME/YOUR-REPO`
5. Branch: `*/main`
6. Script Path: `Jenkinsfile`

### Create Jenkinsfile in GitHub

In your repo, create `Jenkinsfile`:

```groovy
pipeline {
    agent {
        docker { image 'python:3.9' }
    }
    
    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }
        
        stage('Build') {
            steps {
                sh 'python --version'
            }
        }
    }
}
```

### Run

Click "Build Now"

Jenkins will:
1. Pull code from GitHub
2. Find Jenkinsfile
3. Run pipeline

## Part 8: Deploying to Kubernetes

### Understanding CI vs CD

**CI (Continuous Integration) = Jenkins**

What Jenkins does:
- Get code from GitHub
- Build Docker image
- Test code
- Push to Docker Hub
- Update deployment.yaml file

**Jenkins STOPS here!**

**CD (Continuous Deployment) = ArgoCD**

What ArgoCD does:
- Watch GitHub
- See deployment.yaml changed
- Deploy to Kubernetes
- App goes LIVE

**ArgoCD STARTS here!**

**Simple:**
- CI = Build the package (Jenkins)
- CD = Deliver the package (ArgoCD)

### The Complete Flow

**Step 1:** You push code to GitHub

**Step 2:** Jenkins runs (CI Phase)

```groovy
pipeline {
    agent {
        docker { image 'python:3.9' }
    }
    
    environment {
        DOCKER_IMAGE = "your-dockerhub/todo-app"
        BUILD_NUMBER = "${env.BUILD_NUMBER}"
    }
    
    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }
        
        stage('Build Docker Image') {
            steps {
                sh "docker build -t ${DOCKER_IMAGE}:${BUILD_NUMBER} ."
            }
        }
        
        stage('Push to Docker Hub') {
            steps {
                sh "docker push ${DOCKER_IMAGE}:${BUILD_NUMBER}"
            }
        }
        
        stage('Update Kubernetes File') {
            steps {
                sh """
                    sed -i 's|image:.*|image: ${DOCKER_IMAGE}:${BUILD_NUMBER}|' deployment.yaml
                    git add deployment.yaml
                    git commit -m 'Update to build ${BUILD_NUMBER}'
                    git push origin main
                """
            }
        }
    }
}
```

**What each stage does:**

1. **Checkout:** Gets code from GitHub
2. **Build Docker Image:** Creates image (e.g., todo-app:35)
3. **Push to Docker Hub:** Uploads image
4. **Update Kubernetes File:** Changes deployment.yaml from version 1 to 35, pushes to GitHub

**Step 3:** Jenkins finishes ✓ (CI done)

**Step 4:** ArgoCD watches GitHub (CD Phase)

ArgoCD checks every 3 minutes:
- Sees deployment.yaml changed
- Old: `image: todo-app:1`
- New: `image: todo-app:35`
- ArgoCD: "They're different! I need to deploy!"

**Step 5:** ArgoCD deploys

1. Tells Kubernetes: "Use todo-app:35"
2. Kubernetes pulls image from Docker Hub
3. Stops old pods (version 1)
4. Starts new pods (version 35)

**Step 6:** App is LIVE! ✓ (CD done)

### Timeline Example

```
9:00 AM - You push code
9:01 AM - Jenkins builds image, pushes to Docker Hub, updates GitHub
9:04 AM - ArgoCD sees change, deploys to Kubernetes
9:05 AM - App is LIVE!
```

Total: 5 minutes!

### Why This Approach?

**Old way:** Jenkins deploys directly to Kubernetes

**Problems:**
- Jenkins needs Kubernetes credentials (security risk)
- Hard to track what's deployed
- Manual rollback

**New way:** Jenkins → GitHub → ArgoCD → Kubernetes

**Benefits:**
- Jenkins doesn't need Kubernetes access (secure)
- Git shows what's deployed (transparency)
- Easy rollback (revert Git commit)
- Automatic sync

### What Each Tool Does

| Tool | Job |
|------|-----|
| GitHub | Stores code and deployment files |
| Jenkins | Builds image, pushes to Docker Hub |
| Docker Hub | Stores Docker images |
| ArgoCD | Watches GitHub, deploys to Kubernetes |
| Kubernetes | Runs your app |

## Part 9: Common Problems

### Problem 1: Permission Denied

**Error:** `Permission denied while trying to connect to Docker`

**Fix:**
```bash
sudo usermod -aG docker jenkins
sudo systemctl restart docker
sudo systemctl restart jenkins
```

### Problem 2: Docker Not Found

**Error:** `docker: command not found`

**Fix:**
```bash
sudo apt install docker.io -y
sudo systemctl restart jenkins
```

### Problem 3: Can't Access Jenkins

**Error:** Browser can't open Jenkins

**Fix:**
- Check AWS Security Group
- Open port 8080
- Source: 0.0.0.0/0

### Problem 4: Too Many Containers

**Fix:**
```bash
docker stop $(docker ps -aq)
docker rm $(docker ps -aq)
```

### Problem 5: Disk Full

**Fix:**
```bash
docker system prune -a
```

## Part 10: Interview Questions

### Q1: Explain Your CI/CD Process

**Answer:**

"When developer pushes code:
1. Jenkins builds Docker image
2. Pushes to Docker Hub
3. Updates Kubernetes manifest in GitHub
4. ArgoCD sees change and deploys to Kubernetes

We use Docker containers as agents to save cost."

### Q2: Jenkins Master vs Worker?

**Answer:**

"Master schedules jobs, doesn't run builds. Workers run builds. We use Docker containers as workers instead of VMs for cost savings."

### Q3: Why Docker Containers?

**Answer:**

"Containers are:
- Created on-demand
- Deleted when done
- Easy to update (change image version)
- No dependency conflicts
- Cheaper than VMs"

### Q4: Freestyle vs Pipeline?

**Answer:**

"Always use Pipeline:
- Freestyle: Click buttons, can't version control
- Pipeline: Code in Git, easy to share and review"

### Q5: How to Secure Jenkins?

**Answer:**

"1. Strong password
2. Role-based access
3. Credentials plugin for secrets
4. HTTPS
5. Keep updated
6. Restrict access via security groups"

### Q6: What is GitOps?

**Answer:**

"Git is single source of truth. All changes go through Git. ArgoCD watches Git and keeps Kubernetes in sync. Benefits: version control, audit trail, easy rollback."

### Q7: Why ArgoCD Instead of Jenkins for Deployment?

**Answer:**

"ArgoCD follows GitOps. It continuously monitors Git and syncs Kubernetes. Jenkins would need direct Kubernetes access (less secure). ArgoCD provides automatic sync and self-healing."
