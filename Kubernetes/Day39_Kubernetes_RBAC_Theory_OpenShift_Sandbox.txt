================================================================================
            PART 1: WHAT IS KUBERNETES RBAC?
================================================================================

🔹 SIMPLE DEFINITION:

RBAC = Role-Based Access Control
- Control WHO can access WHAT in Kubernetes cluster
- Based on user's ROLE (Developer, QA, Admin)
- Critical for security in production environments

🔹 WHY RBAC IS IMPORTANT:

Without RBAC:
- Anyone can delete critical resources (etcd, API server)
- Developers can access production secrets
- QA team can modify cluster configurations
- Applications can perform unauthorized actions

With RBAC:
- Developers: Can only manage their applications
- QA Team: Can only read resources and logs
- Admins: Can manage entire cluster
- Applications: Limited to required permissions only

🔹 REAL-WORLD EXAMPLE:

Company with Kubernetes cluster:
- 10 DevOps Engineers (need admin access)
- 50 Developers (need app deployment access)
- 20 QA Engineers (need read-only access)
- 100 Applications (need specific API access)

Without RBAC: Everyone has admin access = DISASTER!
With RBAC: Everyone has appropriate access = SECURE!

================================================================================
        PART 2: TWO MAIN AREAS OF RBAC
================================================================================

🔹 AREA 1: USER MANAGEMENT

Who: Human users (developers, QA, admins)
Challenge: How to give different access to different people?

Example scenarios:
- Developer should deploy apps in "dev" namespace only
- QA should read logs but not delete resources
- Admin should manage entire cluster
- Intern should have very limited access

🔹 AREA 2: SERVICE ACCOUNT MANAGEMENT

Who: Applications/Pods running in cluster
Challenge: How to control what applications can do?

Example scenarios:
- Pod should read ConfigMaps but not Secrets
- Monitoring app should read all resources
- Backup app should access storage but not networking
- Malicious pod should not delete cluster resources

🔹 KEY DIFFERENCE:

User Management:
- External people logging into cluster
- Authentication via identity providers
- Temporary access (login/logout)

Service Account Management:
- Applications running inside cluster
- Authentication via service account tokens
- Persistent access (as long as pod runs)

================================================================================
        PART 3: HOW KUBERNETES HANDLES USER AUTHENTICATION
================================================================================

🔹 IMPORTANT CONCEPT:

Kubernetes DOES NOT manage users directly!
Kubernetes offloads user management to Identity Providers.

🔹 WHAT THIS MEANS:

You CANNOT create users in Kubernetes like:
```bash
# This doesn't exist in Kubernetes!
kubectl create user developer
```

Instead, Kubernetes integrates with external systems.

🔹 IDENTITY PROVIDERS KUBERNETES SUPPORTS:

1. **Cloud Provider IAM**
   - AWS IAM (for EKS)
   - Azure AD (for AKS)
   - Google Cloud IAM (for GKE)

2. **Enterprise Identity Systems**
   - LDAP (Active Directory)
   - SAML providers
   - OIDC (OpenID Connect)

3. **Third-party Solutions**
   - Keycloak (popular open-source)
   - Okta
   - Auth0

4. **Social Logins**
   - GitHub
   - Google
   - Microsoft

🔹 HOW IT WORKS (EXAMPLE WITH AWS EKS):

1. Company has AWS account with IAM users
2. EKS cluster configured to use AWS IAM
3. Developer logs in with AWS credentials
4. Kubernetes receives user info from AWS
5. RBAC rules applied based on user identity

This is like "Login with Google" on websites!

🔹 OAUTH INTEGRATION:

Kubernetes API Server acts as OAuth server:
- Receives authentication requests
- Validates with identity provider
- Issues tokens for authorized users
- Applies RBAC rules

🔹 PRODUCTION EXAMPLE:

```
Developer → AWS IAM Login → EKS Cluster → RBAC Rules → Limited Access
Admin → AWS IAM Login → EKS Cluster → RBAC Rules → Full Access
```

================================================================================
        PART 4: SERVICE ACCOUNTS (KUBERNETES MANAGED)
================================================================================

🔹 WHAT ARE SERVICE ACCOUNTS:

Service Accounts = Identities for applications/pods
- Created and managed BY Kubernetes
- Used by pods to access Kubernetes API
- Stored as YAML files (like other Kubernetes resources)

🔹 DEFAULT BEHAVIOR:

Every pod gets a service account automatically:
- If you don't specify: Uses "default" service account
- If you specify: Uses your custom service account

🔹 SERVICE ACCOUNT YAML EXAMPLE:

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app-service-account
  namespace: default
```

Simple as creating a Pod!

🔹 HOW PODS USE SERVICE ACCOUNTS:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  serviceAccountName: my-app-service-account
  containers:
  - name: app
    image: my-app:latest
```

Pod will run with permissions of "my-app-service-account"

================================================================================
        PART 5: THREE COMPONENTS OF RBAC
================================================================================

🔹 COMPONENT 1: USERS/SERVICE ACCOUNTS

Who needs access?
- Users: External people (developers, admins)
- Service Accounts: Applications/pods

🔹 COMPONENT 2: ROLES

What permissions are available?
- Role: Permissions within ONE namespace
- ClusterRole: Permissions across ENTIRE cluster

🔹 COMPONENT 3: ROLE BINDINGS

Who gets which permissions?
- RoleBinding: Assigns Role to users/service accounts
- ClusterRoleBinding: Assigns ClusterRole to users/service accounts

🔹 THE RBAC EQUATION:

```
User/ServiceAccount + Role + RoleBinding = Access Control
```

🔹 SIMPLE ANALOGY:

Think of it like a company:
- **Employee** = User/ServiceAccount (who)
- **Job Description** = Role (what permissions)
- **Employment Contract** = RoleBinding (connects who with what)

================================================================================
        PART 6: ROLES VS CLUSTER ROLES
================================================================================

🔹 ROLE (NAMESPACE-SCOPED):

Permissions within ONE namespace only.

Example Role:
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: development
  name: developer-role
rules:
- apiGroups: [""]
  resources: ["pods", "configmaps"]
  verbs: ["get", "list", "create", "delete"]
```

This role allows:
- Access to pods and configmaps
- Only in "development" namespace
- Cannot access other namespaces

🔹 CLUSTER ROLE (CLUSTER-SCOPED):

Permissions across ENTIRE cluster.

Example ClusterRole:
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: admin-role
rules:
- apiGroups: [""]
  resources: ["*"]
  verbs: ["*"]
```

This role allows:
- Access to ALL resources
- In ALL namespaces
- Full admin permissions

🔹 WHEN TO USE WHAT:

Use Role:
- Developers working on specific projects
- Applications that need namespace-specific access
- Limited scope requirements

Use ClusterRole:
- Cluster administrators
- Monitoring applications (need cluster-wide view)
- System-level applications

================================================================================
        PART 7: ROLE BINDINGS VS CLUSTER ROLE BINDINGS
================================================================================

🔹 ROLE BINDING:

Connects users/service accounts with Roles (namespace-scoped).

Example:
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: developer-binding
  namespace: development
subjects:
- kind: User
  name: john-developer
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: developer-role
  apiGroup: rbac.authorization.k8s.io
```

Result: "john-developer" gets "developer-role" permissions in "development" namespace.

🔹 CLUSTER ROLE BINDING:

Connects users/service accounts with ClusterRoles (cluster-scoped).

Example:
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: admin-binding
subjects:
- kind: User
  name: jane-admin
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: admin-role
  apiGroup: rbac.authorization.k8s.io
```

Result: "jane-admin" gets "admin-role" permissions across entire cluster.

================================================================================
        PART 8: RBAC WORKFLOW EXAMPLE
================================================================================

🔹 SCENARIO: Give developer access to deploy apps

STEP 1: Create Service Account
```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: app-deployer
  namespace: development
```

STEP 2: Create Role
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: development
  name: deployer-role
rules:
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["create", "get", "list", "update", "delete"]
- apiGroups: [""]
  resources: ["pods", "services"]
  verbs: ["get", "list"]
```

STEP 3: Create RoleBinding
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: deployer-binding
  namespace: development
subjects:
- kind: ServiceAccount
  name: app-deployer
  namespace: development
roleRef:
  kind: Role
  name: deployer-role
  apiGroup: rbac.authorization.k8s.io
```

RESULT: "app-deployer" service account can:
- Create/manage deployments in "development" namespace
- View pods and services in "development" namespace
- Cannot access other namespaces
- Cannot delete pods or services

================================================================================
        PART 11: RBAC BEST PRACTICES
================================================================================

🔹 PRINCIPLE OF LEAST PRIVILEGE:

Give minimum permissions required:
- Don't give admin access unless absolutely needed
- Start with read-only access, add permissions as needed
- Regular review and cleanup of permissions

🔹 NAMESPACE ISOLATION:

Use namespaces to separate environments:
- development, staging, production namespaces
- Different teams get access to their namespaces only
- Prevents accidental cross-environment changes

🔹 SERVICE ACCOUNT SECURITY:

- Create specific service accounts for each application
- Don't use default service account for applications
- Regularly rotate service account tokens
- Monitor service account usage

🔹 REGULAR AUDITS:

- Review who has what access
- Remove unused service accounts and roles
- Monitor RBAC-related events
- Document access patterns

        PART 13: INTERVIEW QUESTIONS
================================================================================

❓ **Q1: What is Kubernetes RBAC?**

A: RBAC (Role-Based Access Control) is a security mechanism in Kubernetes that
controls who can access what resources based on their role. It consists of
three components: Users/ServiceAccounts (who), Roles (what permissions), and
RoleBindings (connects who with what).

❓ **Q2: How does Kubernetes handle user authentication?**

A: Kubernetes doesn't manage users directly. It offloads user management to
external identity providers like AWS IAM, LDAP, OIDC, or social logins. The
API server acts as an OAuth server and integrates with these providers.

❓ **Q3: What's the difference between Role and ClusterRole?**

A: Role provides permissions within a single namespace, while ClusterRole
provides permissions across the entire cluster. Use Role for namespace-specific
access and ClusterRole for cluster-wide access.

❓ **Q4: What are Service Accounts?**

A: Service Accounts are identities for applications/pods running in Kubernetes.
They're managed by Kubernetes (unlike users) and are used by pods to access
the Kubernetes API with specific permissions.

❓ **Q5: Why is RBAC important?**

A: RBAC is critical for security. Without it, anyone could delete critical
resources, access sensitive data, or perform unauthorized actions. It
implements the principle of least privilege and prevents security breaches.
