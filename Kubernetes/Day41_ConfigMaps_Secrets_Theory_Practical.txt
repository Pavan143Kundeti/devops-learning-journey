                    PART 1: UNDERSTANDING THE PROBLEM

🔹 REAL-WORLD APPLICATION SCENARIO:

Imagine you have a backend application:
- Application talks to database
- Retrieves information from database
- Sends data back to users

🔹 WHAT INFORMATION DOES APPLICATION NEED?

Application needs database connection details:
✅ Database port (example: 3306)
✅ Database username
✅ Database password  
✅ Connection type
✅ Number of connections
✅ Other configuration settings

🔹 THE PROBLEM WITH HARDCODING:

❌ BAD PRACTICE: Hardcode values in application code
```python
# DON'T DO THIS!
DB_PORT = 3306
DB_USER = "admin"
DB_PASS = "secret123"
```

Why this is bad:
- If database port changes → Application breaks
- If username changes → Application fails
- If password changes → Cannot connect to database
- Need to rebuild application for every change

🔹 GENERAL SOLUTION (Outside Kubernetes):

✅ GOOD PRACTICE: Use environment variables or config files
```python
# DO THIS!
import os
DB_PORT = os.getenv('DB_PORT')
DB_USER = os.getenv('DB_USER')  
DB_PASS = os.getenv('DB_PASS')
```

Read from:
- Environment variables
- Configuration files in filesystem
- External configuration services

================================================================================
                    PART 2: KUBERNETES SOLUTION - CONFIGMAPS
================================================================================

🔹 THE KUBERNETES WAY:

In Kubernetes world, we deal with containers in Pods.

Problem: How to get configuration into containers?
- Cannot manually login to containers
- Cannot create environment variables inside running containers
- Values might change frequently
- Don't know values at Docker build time

🔹 WHAT IS A CONFIGMAP?

ConfigMap = Kubernetes resource to store configuration data

Simple definition:
- Store non-sensitive configuration information
- Make it available to Pods as environment variables or files
- Separate configuration from application code

🔹 HOW CONFIGMAPS WORK:

**Step 1:** DevOps Engineer creates ConfigMap
```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  DB_PORT: "3306"
  DB_CONNECTION_TYPE: "mysql"
  MAX_CONNECTIONS: "100"
```

**Step 2:** Pod uses ConfigMap data
- As environment variables
- As mounted files

**Step 3:** Application reads configuration
- From environment variables
- From mounted files

🔹 CONFIGMAP ADVANTAGES:

✅ Centralized configuration management
✅ Easy to update without rebuilding images
✅ Separate configuration from code
✅ Version control for configurations
✅ Reusable across multiple applications
================================================================================
                    PART 3: UNDERSTANDING SECRETS
================================================================================

🔹 WHY DO WE NEED SECRETS?

If ConfigMaps solve the configuration problem, why do we need Secrets?

The answer: SENSITIVE DATA

🔹 THE SECURITY PROBLEM:

When you create any Kubernetes resource:
- Data gets stored in etcd (Kubernetes database)
- Data is stored as plain objects in etcd
- If hacker gets access to etcd → All data is compromised

Example with ConfigMap storing sensitive data:
```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: db-config
data:
  DB_PORT: "3306"
  DB_USERNAME: "admin"      # ← SENSITIVE!
  DB_PASSWORD: "secret123"  # ← SENSITIVE!
```

Problem: Hacker can read username/password from etcd!

🔹 WHAT IS A SECRET?

Secret = Kubernetes resource for storing SENSITIVE data

Key difference from ConfigMap:
- ConfigMap: Non-sensitive data (ports, URLs, feature flags)
- Secret: Sensitive data (passwords, API keys, certificates)

🔹 HOW SECRETS SOLVE SECURITY PROBLEM:

**Encryption at Rest:**
- Kubernetes encrypts Secret data before storing in etcd
- Even if hacker accesses etcd, data is encrypted
- Hacker cannot read encrypted data without decryption key

**RBAC (Role-Based Access Control):**
- Restrict who can access Secrets
- Developers can read ConfigMaps but not Secrets
- Only DevOps engineers get Secret access
- Principle of least privilege

🔹 CONFIGMAP VS SECRET COMPARISON:

| Aspect | ConfigMap | Secret |
|--------|-----------|---------|
| **Purpose** | Non-sensitive config | Sensitive data |
| **Storage** | Plain text in etcd | Encrypted in etcd |
| **Access** | Broader access allowed | Restricted access |
| **Examples** | Ports, URLs, feature flags | Passwords, API keys, certificates |
| **Security** | Basic | Enhanced with encryption + RBAC |

🔹 INTERVIEW ANSWER TEMPLATE:

"Both ConfigMaps and Secrets store configuration data for applications in Kubernetes. The key difference is:

- **ConfigMaps** store non-sensitive information like database ports, API URLs, and feature flags. Data is stored in plain text.

- **Secrets** store sensitive information like passwords, API keys, and certificates. Data is encrypted at rest in etcd.

Secrets also support stronger RBAC controls to limit access to sensitive data."

            PART 5: USING CONFIGMAP AS ENVIRONMENT VARIABLES
================================================================================

--------------------------------------------------------------------------------
STEP 1: MODIFY DEPLOYMENT TO USE CONFIGMAP
--------------------------------------------------------------------------------

Edit `deployment.yaml` to add environment variables:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: python-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: python-app
  template:
    metadata:
      labels:
        app: python-app
    spec:
      containers:
      - name: python-app
        image: abhishekf5/python-django-app:latest
        ports:
        - containerPort: 8000
        env:                           # ← ADD THIS SECTION
        - name: DB_PORT               # Environment variable name
          valueFrom:
            configMapKeyRef:
              name: test-cm           # ConfigMap name
              key: DB-PORT            # Key from ConfigMap
```

🔹 EXPLANATION OF NEW SECTION:

- `env:` → List of environment variables
- `name: DB_PORT` → Environment variable name in container
- `valueFrom:` → Get value from external source
- `configMapKeyRef:` → Reference to ConfigMap
- `name: test-cm` → ConfigMap name we created
- `key: DB-PORT` → Key from ConfigMap data section

--------------------------------------------------------------------------------


- New pods get the ConfigMap data as environment variables

        PART 6: PROBLEM WITH ENVIRONMENT VARIABLES
================================================================================

🔹 THE LIMITATION:

Environment variables cannot be changed in running containers!

Let's demonstrate this problem:

--------------------------------------------------------------------------------
STEP 1: CHANGE CONFIGMAP VALUE
--------------------------------------------------------------------------------

Edit `cm.yaml`:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: test-cm
data:
  DB-PORT: "3307"  # ← Changed from 3306 to 3307
```

Apply the change:
```bash
kubectl apply -f cm.yaml
```

Verify ConfigMap is updated:
```bash
kubectl describe cm test-cm
```

You'll see the new value: `3307`

--------------------------------------------------------------------------------
STEP 2: CHECK IF POD ENVIRONMENT VARIABLE UPDATED
--------------------------------------------------------------------------------

```bash
# Get pod name
kubectl get pods

# Exec into pod
kubectl exec -it <pod-name> -- /bin/bash

# Check environment variable
env | grep DB
```

Result: Still shows `DB_PORT=3306` (old value)!

🔹 THE PROBLEM:

- ConfigMap updated successfully
- But Pod environment variable didn't change
- Application still uses old database port
- This will cause connection failures!

🔹 WHY THIS HAPPENS:

- Environment variables are set when container starts
- Cannot modify environment variables in running containers
- Only way to update: Restart the entire Pod/Deployment
- In production: Restarting causes downtime and traffic loss

🔹 THE SOLUTION:

Use Volume Mounts instead of Environment Variables!
================================================================================
            PART 7: USING CONFIGMAP AS VOLUME MOUNTS
================================================================================

🔹 WHY VOLUME MOUNTS?

Volume mounts solve the environment variable limitation:
- ConfigMap data becomes files in Pod filesystem
- Files can be updated automatically when ConfigMap changes
- No need to restart Pods for configuration updates
- Application reads from files instead of environment variables

--------------------------------------------------------------------------------
STEP 1: MODIFY DEPLOYMENT FOR VOLUME MOUNTS
--------------------------------------------------------------------------------

Edit `deployment.yaml` to use volume mounts:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: python-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: python-app
  template:
    metadata:
      labels:
        app: python-app
    spec:
      containers:
      - name: python-app
        image: abhishekf5/python-django-app:latest
        ports:
        - containerPort: 8000
        volumeMounts:                    # ← ADD THIS SECTION
        - name: db-connection           # Volume name (must match below)
          mountPath: /opt               # Where to mount in container
      volumes:                          # ← ADD THIS SECTION
      - name: db-connection            # Volume name
        configMap:
          name: test-cm                 # ConfigMap name
```

🔹 EXPLANATION OF NEW SECTIONS:

**volumeMounts:**
- `name: db-connection` → Volume name (connects to volumes section)
- `mountPath: /opt` → Directory in container where files appear

**volumes:**
- `name: db-connection` → Volume name (connects to volumeMounts)
- `configMap.name: test-cm` → Source ConfigMap

🔹 WHAT HAPPENS:

1. Kubernetes creates volume from ConfigMap
2. Volume gets mounted at `/opt` in container
3. ConfigMap keys become filenames
4. ConfigMap values become file contents
5. Example: `DB-PORT` key → `/opt/DB-PORT` file with content `3307`

--------------------------------------------------------------------------------
STEP 2: APPLY UPDATED DEPLOYMENT
--------------------------------------------------------------------------------

```bash
kubectl apply -f deployment.yaml
```

🔹 VERIFY PODS ARE UPDATED:

```bash
kubectl get pods -w
```

Wait for new pods to be `Running`.

--------------------------------------------------------------------------------
STEP 3: VERIFY VOLUME MOUNT IN POD
--------------------------------------------------------------------------------

🔹 EXEC INTO POD:

```bash
kubectl get pods
kubectl exec -it <pod-name> -- /bin/bash
```

🔹 CHECK MOUNTED FILES:

```bash

```

Expected output: `3307`

Success! ConfigMap data is now available as files.

🔹 HOW APPLICATION READS THIS:

```python

    db_port = f.read().strip()
    
print(f"Database port: {db_port}")
```

                PART 8: WORKING WITH SECRETS
================================================================================

🔹 CREATING SECRETS - COMMAND LINE METHOD:

Instead of YAML file, you can create Secrets using kubectl:

```bash
kubectl create secret generic test-secret \
  --from-literal=DB-PASSWORD=secret123
```

🔹 COMMAND EXPLANATION:

- `create secret generic` → Create a generic secret
- `test-secret` → Secret name
- `--from-literal` → Create from literal value
- `DB-PASSWORD=secret123` → Key=Value pair

🔹 VERIFY SECRET CREATION:

```bash
kubectl get secrets
kubectl describe secret test-secret
```

Output shows:
```
Name:         test-secret
Type:         Opaque
Data
====
DB-PASSWORD:  10 bytes
```

Notice: Shows byte count, not actual value (security feature)

--------------------------------------------------------------------------------
UNDERSTANDING SECRET ENCRYPTION
--------------------------------------------------------------------------------

🔹 VIEW SECRET IN KUBERNETES:

```bash
kubectl edit secret test-secret
```

You'll see something like:
```yaml
apiVersion: v1
data:
  DB-PASSWORD: c2VjcmV0MTIz  # ← Base64 encoded
kind: Secret
```

🔹 DECODE THE SECRET:

```bash
echo "c2VjcmV0MTIz" | base64 --decode
```

Output: `secret123`

🔹 IMPORTANT SECURITY NOTES:

- Kubernetes uses Base64 encoding by default (NOT strong encryption)
- Base64 is encoding, not encryption (easily reversible)
- For production: Use external secret management (HashiCorp Vault, Sealed Secrets)
- For etcd encryption: Configure encryption keys at API server level

🔹 PRODUCTION SECRET SECURITY:

**Better Options:**
- **HashiCorp Vault** → Enterprise secret management
- **Sealed Secrets** → Encrypt secrets in Git repositories  
- **External Secrets Operator** → Sync from external secret stores
- **AWS Secrets Manager** → Cloud-native secret management
- **Azure Key Vault** → Azure secret management

--------------------------------------------------------------------------------
USING SECRETS IN PODS
--------------------------------------------------------------------------------

🔹 METHOD 1: AS ENVIRONMENT VARIABLES

Add to `deployment.yaml`:
```yaml
env:
- name: DB_PASSWORD
  valueFrom:
    secretKeyRef:           # ← secretKeyRef instead of configMapKeyRef
      name: test-secret
      key: DB-PASSWORD
```

🔹 METHOD 2: AS VOLUME MOUNTS

Add to `deployment.yaml`:
```yaml
volumeMounts:
- name: secret-volume
  mountPath: /etc/secrets
volumes:
- name: secret-volume
  secret:                   # ← secret instead of configMap
    secretName: test-secret
```

================================================================================
                PART 9: DIFFERENT TYPES OF SECRETS
================================================================================

🔹 GENERIC SECRETS:

For general key-value pairs:
```bash
kubectl create secret generic my-secret \
  --from-literal=username=admin \
  --from-literal=password=secret123
```

🔹 TLS SECRETS:

For storing SSL/TLS certificates:
```bash
kubectl create secret tls my-tls-secret \
  --cert=path/to/cert.crt \
  --key=path/to/cert.key
```

Use case: HTTPS certificates for Ingress

🔹 DOCKER REGISTRY SECRETS:

For pulling images from private registries:
```bash
kubectl create secret docker-registry my-registry-secret \
  --docker-server=registry.example.com \
  --docker-username=myuser \
  --docker-password=mypass \
  --docker-email=myemail@example.com
```

Use case: Pull images from private Docker Hub or AWS ECR

🔹 SERVICE ACCOUNT SECRETS:

Automatically created for service accounts:
- Used for Pod-to-API server authentication
- Mounted automatically at `/var/run/secrets/kubernetes.io/serviceaccount/`

                PART 11: BEST PRACTICES
================================================================================

🔹 CONFIGMAP BEST PRACTICES:

✅ **DO:**
- Use for non-sensitive configuration
- Group related configurations together
- Use meaningful names
- Version your ConfigMaps
- Document configuration keys

❌ **DON'T:**
- Store sensitive data in ConfigMaps
- Make ConfigMaps too large (1MB limit)
- Use ConfigMaps for binary data

🔹 SECRET BEST PRACTICES:

✅ **DO:**
- Use for sensitive data only
- Implement strong RBAC controls
- Use external secret management in production
- Rotate secrets regularly
- Monitor secret access

❌ **DON'T:**
- Store secrets in Git repositories
- Use default base64 encoding for production
- Give broad access to secrets
- Log secret values

🔹 VOLUME MOUNT VS ENVIRONMENT VARIABLES:

**Use Volume Mounts when:**
- Configuration changes frequently
- Need automatic updates without Pod restart
- Large configuration files
- Multiple configuration files

**Use Environment Variables when:**
- Simple key-value pairs
- Configuration rarely changes
- Application expects environment variables
- Backward compatibility needed

                PART 13: INTERVIEW QUESTIONS
================================================================================

❓ **Q1: What's the difference between ConfigMap and Secret?**

A: Both store configuration data, but ConfigMaps store non-sensitive data in plain text, while Secrets store sensitive data with base64 encoding and encryption at rest in etcd. Secrets also support stronger RBAC controls.

❓ **Q2: How do you update configuration without restarting Pods?**

A: Use volume mounts instead of environment variables. When you update a ConfigMap or Secret, mounted files automatically update in 30-60 seconds without Pod restart.

❓ **Q3: What are the different ways to use ConfigMaps in Pods?**

A: Two main ways:
1. Environment variables using `configMapKeyRef`
2. Volume mounts using `configMap` volume type

❓ **Q4: How do you create a Secret from command line?**

A: Use `kubectl create secret generic <name> --from-literal=key=value` for generic secrets, or specific commands for TLS and Docker registry secrets.

❓ **Q5: Is base64 encoding in Secrets secure?**

A: No, base64 is encoding, not encryption. For production, use external secret management like HashiCorp Vault or configure etcd encryption keys.

❓ **Q6: What happens if you delete a ConfigMap that's being used by a Pod?**

A: If using environment variables, Pod continues with existing values. If using volume mounts, mounted files become unavailable and application may fail.

                PART 15: KEY TAKEAWAYS
================================================================================

✅ **ConfigMaps:**
- Store non-sensitive configuration data
- Use for ports, URLs, feature flags, application settings
- Data stored in plain text in etcd
- Can be used as environment variables or volume mounts

✅ **Secrets:**
- Store sensitive data like passwords, API keys, certificates
- Data is base64 encoded and encrypted at rest in etcd
- Support stronger RBAC controls
- Use external secret management for production

✅ **Volume Mounts vs Environment Variables:**
- Volume mounts: Automatic updates, no Pod restart needed
- Environment variables: Simple, but require Pod restart for updates

✅ **Best Practices:**
- Use appropriate resource type (ConfigMap vs Secret)
- Implement proper RBAC for Secrets
- Use external secret management in production
- Prefer volume mounts for frequently changing configuration

✅ **Security:**
- Never store sensitive data in ConfigMaps
- Use strong RBAC controls for Secrets
- Consider external secret management solutions
- Rotate secrets regularly
