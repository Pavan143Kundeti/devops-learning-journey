================================================================================
            PART 1: UNDERSTANDING THE NEED FOR CUSTOM RESOURCES
================================================================================

🔹 KUBERNETES NATIVE RESOURCES:

Out-of-the-box Kubernetes resources:
- Pod, Deployment, Service
- ConfigMap, Secret
- Ingress, PersistentVolume
- Namespace, Node

These cover basic container orchestration needs.

🔹 THE LIMITATION:

What if you need features that native Kubernetes doesn't provide?

Examples:
- Advanced security scanning (Kube-hunter, Kyverno)
- GitOps capabilities (ArgoCD, Flux)
- Service mesh (Istio, Linkerd)
- Monitoring (Prometheus Operator)
- Identity management (Keycloak)
- API Gateway features
- Custom backup solutions
- Advanced networking

🔹 THE CHALLENGE:

Kubernetes cannot build-in support for every possible feature because:
- Thousands of different requirements
- New tools emerge constantly
- Different organizations need different capabilities
- Impossible to maintain all features in core Kubernetes

🔹 THE SOLUTION:

Kubernetes provides a way to EXTEND its API:
- Add new resource types
- Define custom behaviors
- Integrate third-party tools
- Maintain separation of concerns

This is done through Custom Resources!

================================================================================
        PART 2: THREE COMPONENTS OF CUSTOM RESOURCES
================================================================================

🔹 THE THREE ACTORS:

1. **Custom Resource Definition (CRD)**
   - Defines new API type
   - Like a "template" or "schema"
   - Created by tool developers

2. **Custom Resource (CR)**
   - Instance of the CRD
   - Like a "deployment.yaml" but for custom type
   - Created by users

3. **Custom Controller**
   - Watches for Custom Resources
   - Implements the actual functionality
   - Created by tool developers

🔹 ANALOGY WITH NATIVE RESOURCES:

Native Kubernetes:
```
Resource Definition (built-in) → User creates Deployment → Controller acts
```

Custom Resources:
```
CRD (user-defined) → User creates CR → Custom Controller acts
```

🔹 ROLES AND RESPONSIBILITIES:

**DevOps Engineer:**
- Installs CRDs
- Installs Custom Controllers
- Manages cluster-wide custom resources

**Developer/User:**
- Creates Custom Resources
- Uses the new capabilities
- Follows documentation provided by tool creators

        PART 3: DETAILED EXPLANATION WITH EXAMPLES
================================================================================

🔹 EXAMPLE: ISTIO SERVICE MESH

Let's understand with Istio (popular service mesh tool):

**STEP 1: Problem**
- Kubernetes doesn't have built-in service mesh
- Need advanced traffic management
- Need mutual TLS between services
- Need advanced load balancing

**STEP 2: Solution with Custom Resources**

Istio provides:
- CRDs for VirtualService, DestinationRule, etc.
- Custom Controllers to implement service mesh
- Users create Custom Resources to configure mesh

🔹 THE WORKFLOW:

**Phase 1: DevOps Engineer Setup**
```
1. Install Istio CRDs → Extends Kubernetes API
2. Install Istio Controllers → Implements functionality
3. Cluster now supports Istio resources
```

**Phase 2: User Usage**
```
1. User creates VirtualService (Custom Resource)
2. Istio Controller detects new VirtualService
3. Controller configures service mesh accordingly
```

🔹 COMPARISON WITH DEPLOYMENT:

**Native Deployment:**
```yaml
apiVersion: apps/v1          # Built-in API
kind: Deployment             # Built-in resource type
metadata:
  name: my-app
spec:
  replicas: 3
  # ... rest of deployment spec
```

**Custom Resource (Istio VirtualService):**
```yaml
apiVersion: networking.istio.io/v1alpha3  # Custom API
kind: VirtualService                      # Custom resource type
metadata:
  name: my-service
spec:
  http:
  - route:
    - destination:
        host: my-service
  # ... rest of VirtualService spec
```

================================================================================
        PART 4: HOW CUSTOM RESOURCE DEFINITIONS WORK
================================================================================

🔹 WHAT IS A CRD:

CRD = Template that defines:
- What fields are allowed in Custom Resource
- What values are valid
- Required vs optional fields
- Data types and validation rules

🔹 CRD VALIDATION:

Just like Kubernetes validates your deployment.yaml:
```bash
# This fails because "replicas" should be a number
kubectl apply -f deployment.yaml
# Error: field "replicas" must be integer
```

CRDs validate Custom Resources:
```bash
# This fails if VirtualService doesn't match CRD schema
kubectl apply -f virtualservice.yaml
# Error: field "destination" is required
```

🔹 EXAMPLE CRD STRUCTURE:

```yaml
apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: virtualservices.networking.istio.io
spec:
  group: networking.istio.io
  versions:
  - name: v1alpha3
    served: true
    storage: true
    schema:
      openAPIV3Schema:
        type: object
        properties:
          spec:
            type: object
            properties:
              http:
                type: array
                items:
                  type: object
                  properties:
                    route:
                      type: array
                      # ... more validation rules
  scope: Namespaced
  names:
    plural: virtualservices
    singular: virtualservice
    kind: VirtualService
```

This CRD defines:
- API group: networking.istio.io
- Resource name: VirtualService
- Allowed fields and their types
- Validation rules

================================================================================
        PART 5: HOW CUSTOM CONTROLLERS WORK
================================================================================

🔹 WHAT CONTROLLERS DO:

Controllers are programs that:
1. **Watch** for changes to specific resources
2. **Compare** desired state vs actual state
3. **Act** to make actual state match desired state

🔹 NATIVE CONTROLLER EXAMPLE:

Deployment Controller:
1. Watches for Deployment resources
2. Creates ReplicaSets when Deployment is created
3. Manages Pod scaling based on replicas field

🔹 CUSTOM CONTROLLER EXAMPLE:

Istio Controller:
1. Watches for VirtualService resources
2. Configures Envoy proxies when VirtualService is created
3. Updates traffic routing based on VirtualService rules

🔹 CONTROLLER ARCHITECTURE:

```
Custom Resource Created
↓
Controller Detects Change (via Kubernetes API)
↓
Controller Reads Resource Specification
↓
Controller Takes Action (configure systems, create resources, etc.)
↓
Controller Updates Resource Status
```

🔹 HOW CONTROLLERS ARE IMPLEMENTED:

Most custom controllers are written in:
- **Go** (most popular - Kubernetes is written in Go)
- Python (using kubernetes-client)
- Java (using fabric8 kubernetes-client)

Controllers use:
- **Kubernetes API** to watch resources
- **Client libraries** (client-go for Go)
- **Controller frameworks** (controller-runtime, kubebuilder)

================================================================================
        PART 6: PRACTICAL EXAMPLE - ISTIO INSTALLATION
================================================================================

🔹 PREREQUISITES:

- Kubernetes cluster (Minikube or cloud)
- Helm installed
- kubectl configured

🔹 STEP 1: ADD ISTIO HELM REPOSITORY

```bash
# Add Istio Helm repository
helm repo add istio https://istio-release.storage.googleapis.com/charts

# Update Helm repositories
helm repo update
```

What this does: Prepares to install Istio components

🔹 STEP 2: CREATE ISTIO NAMESPACE

```bash
# Create namespace for Istio system components
kubectl create namespace istio-system
```

🔹 STEP 3: INSTALL ISTIO BASE (CRDs)

```bash
# Install Istio CRDs
helm install istio-base istio/base -n istio-system
```

What this does:
- Installs Custom Resource Definitions
- Extends Kubernetes API with Istio resource types
- No functionality yet, just API extensions

Verify CRDs:
```bash
kubectl get crd | grep istio
```

You'll see CRDs like:
- virtualservices.networking.istio.io
- destinationrules.networking.istio.io
- gateways.networking.istio.io

🔹 STEP 4: INSTALL ISTIO CONTROL PLANE (CONTROLLERS)

```bash
# Install Istio control plane (controllers)
helm install istiod istio/istiod -n istio-system --wait
```

What this does:
- Installs Istio controllers (istiod)
- Controllers now watch for Istio Custom Resources
- Ready to implement service mesh functionality

Verify installation:
```bash
kubectl get pods -n istio-system
```

Should show istiod pod running.

🔹 STEP 5: CREATE CUSTOM RESOURCE (USER ACTION)

Now users can create Istio resources:

```yaml
# virtualservice.yaml
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: my-app-vs
spec:
  http:
  - route:
    - destination:
        host: my-app-service
```

Apply the Custom Resource:
```bash
kubectl apply -f virtualservice.yaml
```

🔹 STEP 6: VERIFY CUSTOM RESOURCE

```bash
# List VirtualServices (custom resource)
kubectl get virtualservices

# Describe the custom resource
kubectl describe virtualservice my-app-vs
```

🔹 WHAT HAPPENED:

1. DevOps Engineer installed CRDs (extended API)
2. DevOps Engineer installed Controllers (added functionality)
3. User created Custom Resource (VirtualService)
4. Controller detected Custom Resource
5. Controller configured service mesh accordingly

================================================================================
        PART 8: WRITING CUSTOM CONTROLLERS (OVERVIEW)
================================================================================

🔹 WHEN TO WRITE CUSTOM CONTROLLERS:

- Existing tools don't meet your needs
- You have unique business requirements
- You want to automate complex operational tasks
- You need to integrate with proprietary systems

🔹 PROGRAMMING LANGUAGES:

**Go (Most Popular):**
- Kubernetes is written in Go
- Best client library support (client-go)
- Strong community and examples
- Performance benefits

**Python:**
- Easier for beginners
- Good for simple controllers
- kubernetes-client library available

**Java:**
- Enterprise environments
- fabric8 kubernetes-client

🔹 KEY CONCEPTS FOR CONTROLLER DEVELOPMENT:

**1. Watchers:**
- Monitor Kubernetes API for resource changes
- Trigger actions when resources are created/updated/deleted

**2. Work Queues:**
- Handle events asynchronously
- Ensure reliable processing

**3. Reconciliation Loop:**
- Compare desired state (from Custom Resource)
- Compare actual state (from system)
- Take action to align actual with desired

**4. Client Libraries:**
- **client-go**: Official Go client
- **controller-runtime**: Framework for building controllers
- **kubebuilder**: Scaffolding tool for controllers

🔹 CONTROLLER DEVELOPMENT WORKFLOW:

```
1. Define CRD (what resources look like)
2. Generate client code (from CRD)
3. Implement controller logic (watch, queue, reconcile)
4. Handle events (create, update, delete)
5. Update resource status
6. Test and deploy
```

🔹 EXAMPLE CONTROLLER STRUCTURE (GO):

```go
// Simplified controller structure
type Controller struct {
    client     kubernetes.Interface
    workqueue  workqueue.RateLimitingInterface
    informer   cache.SharedIndexInformer
}

func (c *Controller) Run() {
    // Set up event handlers
    c.informer.AddEventHandler(cache.ResourceEventHandlerFuncs{
        AddFunc:    c.handleAdd,
        UpdateFunc: c.handleUpdate,
        DeleteFunc: c.handleDelete,
    })
    
    // Start processing work items
    go c.runWorker()
}

func (c *Controller) handleAdd(obj interface{}) {
    // Add item to work queue
    c.workqueue.Add(obj)
}

func (c *Controller) processWorkItem() {
    // Get item from queue
    // Reconcile desired vs actual state
    // Take action if needed
}
```

================================================================================
        PART 10: TROUBLESHOOTING CUSTOM RESOURCES
================================================================================

🔹 COMMON ISSUES:

**1. CRD Not Found:**
```bash
# Error: the server doesn't have a resource type "virtualservices"
# Solution: Install CRDs first
kubectl get crd | grep <resource-type>
```

**2. Controller Not Running:**
```bash
# Check controller pods
kubectl get pods -n <controller-namespace>

# Check controller logs
kubectl logs -n <controller-namespace> <controller-pod>
```

**3. Custom Resource Validation Errors:**
```bash
# Error: field "xyz" is not known
# Solution: Check CRD schema and fix Custom Resource
kubectl describe crd <crd-name>
```

**4. Controller Not Acting on Resources:**
```bash
# Check resource status
kubectl describe <custom-resource-type> <name>

# Check controller logs for errors
kubectl logs -n <controller-namespace> <controller-pod>
```

🔹 DEBUGGING WORKFLOW:

1. **Verify CRD exists:**
   ```bash
   kubectl get crd | grep <resource-type>
   ```

2. **Check controller is running:**
   ```bash
   kubectl get pods -n <controller-namespace>
   ```

3. **Validate Custom Resource:**
   ```bash
   kubectl describe <custom-resource-type> <name>
   ```

4. **Check controller logs:**
   ```bash
   kubectl logs -n <controller-namespace> <controller-pod>
   ```

5. **Check RBAC permissions:**
   ```bash
   kubectl describe clusterrole <controller-role>
   ```

================================================================================
        PART 11: INTERVIEW QUESTIONS
================================================================================

❓ **Q1: What are Custom Resource Definitions (CRDs)?**

A: CRDs are Kubernetes resources that extend the Kubernetes API by defining
new resource types. They act as schemas that validate Custom Resources and
allow third-party tools to integrate with Kubernetes using native kubectl
commands and API patterns.

❓ **Q2: What's the difference between CRD, CR, and Custom Controller?**

A: CRD defines the schema for new resource types, CR is an instance of that
resource type created by users, and Custom Controller is the program that
watches for CRs and implements the actual functionality. All three are needed
for a complete custom resource implementation.

❓ **Q3: Why would you use Custom Resources instead of ConfigMaps?**

A: Custom Resources provide schema validation, versioning, status tracking,
and integration with kubectl and Kubernetes API. They enable proper resource
lifecycle management and can trigger automated actions through controllers,
while ConfigMaps are just key-value storage.

❓ **Q4: How do you troubleshoot a Custom Resource that's not working?**

A: Check if CRD exists, verify controller is running, validate CR syntax
against CRD schema, check controller logs for errors, verify RBAC permissions,
and use kubectl describe to check resource status and events.

❓ **Q5: What programming languages can you use to write Custom Controllers?**

A: Go is most popular (Kubernetes is written in Go), but you can also use
Python, Java, or other languages. Go has the best support with client-go
library and controller-runtime framework.

        PART 12: REAL-WORLD EXAMPLES

🔹 EXAMPLE 1: ARGOCD GITOPS

**Problem:** Need GitOps continuous delivery
**Solution:** ArgoCD Custom Resources

CRDs installed:
- Application (defines what to deploy)
- AppProject (groups applications)
- ApplicationSet (manages multiple applications)

Custom Resources created:
```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app
spec:
  source:
    repoURL: https://github.com/my-org/my-app
    path: k8s/
  destination:
    server: https://kubernetes.default.svc
    namespace: default
```

🔹 EXAMPLE 2: PROMETHEUS MONITORING

**Problem:** Need monitoring and alerting
**Solution:** Prometheus Operator

CRDs installed:
- Prometheus (monitoring server)
- ServiceMonitor (defines what to monitor)
- PrometheusRule (alerting rules)

Custom Resources created:
```yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: my-app-monitor
spec:
  selector:
    matchLabels:
      app: my-app
  endpoints:
  - port: metrics
```

🔹 EXAMPLE 3: CROSSPLANE INFRASTRUCTURE

**Problem:** Need to manage cloud resources from Kubernetes
**Solution:** Crossplane

CRDs installed:
- Composition (infrastructure templates)
- CompositeResourceDefinition (custom resource types)
- Provider (cloud provider integration)

Custom Resources created:
```yaml
apiVersion: database.example.com/v1alpha1
kind: PostgreSQLInstance
metadata:
  name: my-database
spec:
  parameters:
    storageGB: 20
    version: "13"
```
