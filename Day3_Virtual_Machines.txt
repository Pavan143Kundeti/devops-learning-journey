# DevOps Day 3 – Virtual Machines (Simple Notes)

## 1. First Understand the Main Goal of DevOps

DevOps mainly tries to **increase efficiency**.

Efficiency means:

* Use **less resources**
* Get **more output**
* **Avoid waste**

Example:

If you buy a **big machine but use only 10% of it**, that is waste.

DevOps tries to **fix this waste**.

---

# 2. Real Life Example (Land and House)

Imagine this situation.

You own **1 acre of land**.

On that land you build a **big house** for your family.

Your family uses only **half of the land**.

The remaining **half acre is empty**.

So what is happening?

You are **wasting land**.

---

## Better Idea

Instead of wasting land, you build **another small house** on the empty land.

Then you **rent it to another family**.

Now:

* You still live in your house
* Another family lives in their house
* Both are independent
* Nobody disturbs each other
* You also earn rent

So the same land is now used **more efficiently**.

---

### Important Lesson

Old way

```
1 acre land
1 house
1 family
Half land wasted
```

New way

```
1 acre land
2 houses
2 families
No waste
```

This idea is **exactly what virtualization does in computers**.

---

# 3. Now Convert This Example Into Software World

In software companies we use something called a **server**.

## What is a Server?

A server is simply a **computer that runs applications for users**.

Example:

When you open

* Google
* Amazon
* Netflix
* Instagram

Those websites are running on **servers somewhere in the world**.

Your laptop sends a request to the **server**.

The server sends back the website.

---

# 4. Problem with Old Servers

Suppose a company buys **5 physical servers**.

Example:

```
Server 1
Server 2
Server 3
Server 4
Server 5
```

Each server may have:

* 100 GB RAM
* 100 CPU cores

Now suppose a team deploys an application.

That application only needs:

```
4 GB RAM
4 CPU cores
```

But the full server has:

```
100 GB RAM
100 CPU cores
```

So what happens?

```
Used: 4 GB
Unused: 96 GB
```

Huge waste.

This is called **underutilization**.

---

# 5. Real Company Problem

Imagine 5 teams in the company.

They are given 5 servers.

But each team only uses **10–20% of server capacity**.

So:

```
80–90% resources wasted
```

Servers are **very expensive**.

Companies cannot keep wasting them.

So engineers needed a better solution.

---

# 6. The Solution: Virtualization

Virtualization means:

**Divide one physical server into many smaller virtual servers.**

Instead of:

```
1 physical server → 1 user
```

We do:

```
1 physical server → many virtual servers
```

---

# 7. What is a Virtual Machine?

A **Virtual Machine (VM)** is a **software-based computer**.

It behaves exactly like a real computer.

Inside a VM you can install:

* Linux
* Windows
* Software
* Databases
* Applications

But it is **not a physical computer**.

It exists **only in software**.

---

# 8. How It Works

Suppose we have **one physical server**.

Instead of giving the whole server to one team, we divide it.

Example:

```
Physical Server
     |
     |
Hypervisor
     |
---------------------------------
| VM1 | VM2 | VM3 | VM4 | VM5 |
---------------------------------
```

Each VM works like a **separate computer**.

Team usage example:

```
Team 1 → VM1
Team 2 → VM2
Team 3 → VM3
Team 4 → VM4
Team 5 → VM5
```

Now the same server is used by **many teams**.

Efficiency improved.

---

# 9. What is a Hypervisor?

A **Hypervisor** is the software that **creates and manages virtual machines**.

Think of it like a **manager**.

It controls:

* CPU
* RAM
* Storage
* Network

And divides them between VMs.

---

## Example Hypervisors

Popular hypervisors:

* VMware
* Xen
* KVM
* Hyper-V

---

# 10. Important Feature of Virtual Machines

Each VM behaves like a **real independent computer**.

Each VM has its own:

```
CPU
RAM
Storage
Operating System
Network
```

Even though they are on the **same physical machine**.

Example:

```
VM1 → Ubuntu
VM2 → Windows
VM3 → CentOS
```

All running on the same physical server.

---

# 11. Real World Example – AWS

Now let’s see how **cloud platforms** use this.

Example cloud providers:

* AWS
* Google Cloud
* Microsoft Azure

These companies build **huge data centers**.

Inside data centers they keep **thousands of physical servers**.

Example location:

```
Mumbai
Singapore
Ohio
London
Tokyo
```

Each location is called a **region**.

---

# 12. What Happens When You Create a VM in AWS

Suppose you go to AWS and request:

```
RAM: 10 GB
CPU: 12 cores
Region: Mumbai
```

Steps that happen:

Step 1
AWS receives your request.

Step 2
AWS finds a physical server in the **Mumbai data center**.

Step 3
That server has a **hypervisor** installed.

Step 4
Hypervisor creates a **virtual machine** for you.

Step 5
AWS gives you:

```
IP Address
Login Key
Access credentials
```

Now you can connect to that VM.

---

# 13. Important Point

You **do not own the physical machine**.

You only get **remote access**.

You cannot go to AWS and say:

"Show me my server."

Because your VM is just a **virtual part of a big physical server**.

---

# 14. Why Virtual Machines Changed the Industry

Before virtualization:

```
1 server = 1 customer
```

After virtualization:

```
1 server = 100+ customers
```

This allowed cloud companies to serve **millions of users**.

That is why virtualization became **a revolution in IT**.

---

# 15. Simple Laptop Example

Even your laptop can run VMs.

Example software:

* Oracle VirtualBox
* VMware Workstation

You can run:

```
Laptop (Windows)
    |
Virtual Machine (Ubuntu)
```

Both run at the same time.

Your laptop is the **physical machine**.

Ubuntu inside VirtualBox is the **virtual machine**.

---

# 16. Final Simple Definition

Virtual Machine =

A **software computer created inside another computer** using virtualization technology.

It behaves exactly like a real computer but **exists only logically, not physically**.

---

# 17. One-Line Memory Trick

Physical Server → Real Computer
Hypervisor → Manager that divides resources
Virtual Machine → Software computer created from the server

---

# 18. DevOps Connection

DevOps engineers use VMs to:

* Deploy applications
* Test environments
* Run CI/CD pipelines
* Host services
* Scale infrastructure

Almost every cloud infrastructure is based on **virtual machines**.
