## Part 1: Why Containers Are Lightweight (Deep Dive)

### Quick Recap from Day 23:

**Virtual Machine:**
- Has FULL operating system
- Heavy (2-5 GB)
- Slow to start (1-2 minutes)

**Container:**
- Has MINIMAL operating system
- Lightweight (100-500 MB)
- Fast to start (1-2 seconds)

### But WHY Are Containers Lightweight?

Let's understand this in detail!



### The Secret: Resource Sharing

**Key Concept:** Containers share resources from the host operating system!

```
┌─────────────────────────────────────────────┐
│         Host Operating System               │
│              (Has Kernel)                   │
└──────────────┬──────────────────────────────┘
               ↓
    ┌──────────┴──────────┐
    ↓                     ↓
┌─────────┐          ┌─────────┐
│ Container 1│          │Container 2│
│           │          │           │
│ Minimal OS│          │ Minimal OS│
│ App 1     │          │ App 2     │
└─────────┘          └─────────┘

Both containers share kernel from host!
```

### What Containers Have vs What They Share

#### Files/Folders INSIDE Container (Logical Isolation):

**These create separation between containers:**

1. **/bin** - Binary execution files
   - Container-specific binaries
   - Application executables

2. **/sbin** - System binary files
   - System-level executables
   - Admin commands

3. **/etc** - Configuration files
   - Application configs
   - Service settings

4. **/lib** - Library files
   - Application libraries
   - Shared libraries

5. **/usr** - User-related files
   - User programs
   - User data

6. **/var** - Variable files
   - Log files
   - Temporary data

7. **/root** - Root home directory
   - Root user files

**Why These Are Needed:**
- Create logical isolation
- Prevent Container A from accessing Container B
- Security boundary
- Each container has its own environment



#### Resources SHARED from Host OS:

**Containers use these from host:**

1. **Host File System**
   - Underlying storage system
   - Disk I/O operations

2. **Networking Stack**
   - Network interfaces
   - TCP/IP stack
   - Network drivers

3. **Kernel**
   - Core operating system
   - System calls
   - Process management

4. **Namespaces**
   - Process isolation
   - Network isolation
   - Mount isolation

5. **Control Groups (cgroups)**
   - Resource limits
   - CPU allocation
   - Memory allocation

**Why Sharing is Good:**
- Saves disk space
- Saves memory
- Faster startup
- Better resource utilization

### Security: Logical Isolation Explained

**The Problem Without Isolation:**

```
If containers shared EVERYTHING:

Hacker enters Container A
         ↓
Can access Container B files
         ↓
Can access Container C files
         ↓
Entire system compromised!
```

**The Solution With Isolation:**

```
Hacker enters Container A
         ↓
Cannot access Container B
         ↓
Only Container A is compromised
         ↓
Other containers are safe!
```

**How It Works:**
- Each container has own /bin, /etc, /lib
- Container A cannot read Container B's files
- Kernel enforces this separation
- Namespaces provide isolation



### Real Example: Ubuntu Container vs Ubuntu VM

#### Ubuntu Virtual Machine:
```
Size: 2.3 GB
Contains:
- Full Linux kernel
- All system files
- All libraries
- GUI components
- System services
- Everything!
```

#### Ubuntu Container (Official Docker Image):
```
Size: 28.16 MB (on Linux AMD64)
Contains:
- Minimal base image
- Essential binaries
- Basic libraries
- Configuration files

Uses from host:
- Kernel
- File system
- Networking
```

**Comparison:**
```
VM:        2,300 MB
Container:   28 MB

Container is 82x smaller!
```

### Why This Matters in Real World

**Scenario: Running 10 Applications**

**With Virtual Machines:**
```
10 VMs × 2 GB each = 20 GB disk space
10 VMs × 512 MB RAM each = 5 GB RAM minimum
Boot time: 10-20 minutes for all VMs
Cost: High
```

**With Containers:**
```
10 Containers × 100 MB each = 1 GB disk space
10 Containers × 50 MB RAM each = 500 MB RAM
Boot time: 10-20 seconds for all containers
Cost: Very low
```

**Savings:**
- 95% less disk space
- 90% less RAM
- 60x faster startup
- Much cheaper!

### Important Interview Point

**Question:** "Why are containers lightweight?"

**Perfect Answer:**
"Containers are lightweight because they don't have a full operating system. They contain only minimal system dependencies needed for logical isolation (like /bin, /etc, /lib directories). The rest of the resources like kernel, file system, and networking stack are shared from the host operating system. This sharing reduces the container size from gigabytes to megabytes while still maintaining security through namespace isolation."

## Part 2: Docker Architecture

### What is Docker?

**Simple Definition:**
Docker is a platform that implements containerization.

**Detailed Definition:**
Docker is a containerization platform that provides an easy way to containerize your applications. It allows you to build container images, run containers, and share images with others.

**Key Point:**
- Containerization = Concept
- Docker = Platform that implements the concept

### Docker vs Other Tools

**Docker Competitors:**
- Podman (addresses Docker's limitations)
- Buildah (we'll learn this too!)
- Skopeo
- CRI-O

**Why Start with Docker?**
- Most popular
- Easy to learn
- Large community
- Lots of tutorials
- Easy troubleshooting

**Later:** Learn Podman and Buildah for advanced use cases

### Docker Architecture Diagram

```
┌─────────────────────────────────────────────────┐
│           Docker Client (CLI)                   │
│                                                 │
│  User runs commands:                            │
│  - docker build                                 │
│  - docker run                                   │
│  - docker pull                                  │
│  - docker push                                  │
└────────────────┬────────────────────────────────┘
                 ↓
                 ↓ (Commands sent via API)
                 ↓
┌─────────────────────────────────────────────────┐
│           Docker Daemon (dockerd)               │
│                                                 │
│  - Receives commands from client                │
│  - Builds images                                │
│  - Runs containers                              │
│  - Manages containers                           │
│  - Communicates with registry                   │
└────┬──────────┬──────────┬──────────────────────┘
     ↓          ↓          ↓
┌─────────┐┌─────────┐┌─────────┐
│ Image 1 ││ Image 2 ││ Image 3 │
└────┬────┘└────┬────┘└────┬────┘
     ↓          ↓          ↓
┌─────────┐┌─────────┐┌─────────┐
│Container││Container││Container│
│    1    ││    2    ││    3    │
└─────────┘└─────────┘└─────────┘
```

### Docker Components Explained

#### 1. Docker Client (CLI)
**What:** Command-line interface you use

**Examples:**
```bash
docker build    # Build image
docker run      # Run container
docker pull     # Download image
docker push     # Upload image
docker ps       # List containers
docker images   # List images
```

**Your Role:** You type commands here



#### 2. Docker Daemon (dockerd)
**What:** Background service that does all the work

**Responsibilities:**
- Listens to Docker API requests
- Builds images from Dockerfiles
- Creates and manages containers
- Manages networks
- Manages volumes (storage)
- Communicates with Docker Registry

**Important:** Docker Daemon is the HEART of Docker!

**Critical Point:**
- If Docker Daemon crashes → All containers stop
- This is called "Single Point of Failure" (SPOF)
- One of Docker's limitations

#### 3. Docker Registry
**What:** Storage for Docker images

**Types:**
- **Public Registry:** Docker Hub (anyone can access)
- **Private Registry:** Your company's registry

**Popular Registries:**
- Docker Hub (hub.docker.com)
- Amazon ECR
- Google Container Registry
- Azure Container Registry
- Quay.io

### How Docker Works (Step by Step)

**Step 1: User Types Command**
```bash
docker build -t myapp:1.0 .
```

**Step 2: Docker Client Sends to Daemon**
```
Client → API Call → Daemon
```

**Step 3: Daemon Processes Request**
```
Daemon reads Dockerfile
Daemon builds image
Daemon tags image as "myapp:1.0"
```

**Step 4: Image is Ready**
```
Image stored locally
Ready to run as container
```

**Step 5: User Runs Container**
```bash
docker run myapp:1.0
```

**Step 6: Daemon Creates Container**
```
Daemon creates container from image
Container starts running
Application is live!
```

## Part 3: Docker Lifecycle

### The Three Stages

```
Stage 1: Dockerfile
    ↓
Stage 2: Docker Image
    ↓
Stage 3: Docker Container
```

### Stage 1: Dockerfile

**What:** Text file with instructions

**Purpose:** Tell Docker how to build your image

**Example:**
```dockerfile
FROM ubuntu:20.04
RUN apt-get update
RUN apt-get install python3
COPY app.py /app/
CMD ["python3", "/app/app.py"]
```

**Think of it as:** Recipe for cooking



### Stage 2: Docker Image

**What:** Template/blueprint for containers

**Created by:** Running `docker build` command

**Characteristics:**
- Read-only
- Can be shared
- Can be versioned
- Stored locally or in registry

**Think of it as:** Snapshot of your application

**Command to Create:**
```bash
docker build -t myapp:1.0 .
```

### Stage 3: Docker Container

**What:** Running instance of an image

**Created by:** Running `docker run` command

**Characteristics:**
- Running application
- Has its own filesystem
- Has its own network
- Can be started/stopped
- Can be deleted

**Think of it as:** Actual running application

**Command to Create:**
```bash
docker run myapp:1.0
```

### Complete Lifecycle Example

```
1. Write Dockerfile
   ↓
   FROM ubuntu:20.04
   RUN apt-get install python3
   COPY app.py /app/
   CMD ["python3", "/app/app.py"]

2. Build Image
   ↓
   docker build -t myapp:1.0 .
   ↓
   Image created: myapp:1.0

3. Run Container
   ↓
   docker run myapp:1.0
   ↓
   Container running!
   Application is live!

4. Share Image (Optional)
   ↓
   docker push myapp:1.0
   ↓
   Image on Docker Hub
   Others can use it!
```

### Why This Workflow is Powerful

**Traditional Way (Without Docker):**
```
1. Create VM
2. Install OS
3. Install Python
4. Install dependencies
5. Copy application
6. Configure everything
7. Run application

Time: 30-60 minutes
Error-prone: Yes
Repeatable: Difficult
```

**Docker Way:**
```
1. Write Dockerfile (once)
2. docker build
3. docker run

Time: 2-5 minutes
Error-prone: No
Repeatable: Always!
```

## Part 5: Installing Docker

### Prerequisites

**You Need:**
- EC2 instance (Ubuntu recommended)
- OR Virtual Machine
- OR Local Linux machine

**For Windows/Mac Users:**
- Install Docker Desktop instead
- Download from: docker.com/products/docker-desktop

### Installation on Ubuntu EC2

#### Step 1: Create EC2 Instance

**Specifications:**
- OS: Ubuntu 20.04 or 22.04
- Instance Type: t2.micro (free tier)
- Storage: 8 GB minimum
- Security Group: Allow SSH (port 22)

#### Step 2: Connect to EC2

```bash
ssh -i your-key.pem ubuntu@your-ec2-ip
```



#### Step 3: Update Package Repository

```bash
sudo apt update
```

**What this does:** Updates list of available packages

#### Step 4: Install Docker

```bash
sudo apt install docker.io -y
```

**What this does:**
- Downloads Docker
- Installs Docker
- Sets up Docker daemon

**Time:** 1-2 minutes

#### Step 5: Verify Installation

```bash
sudo systemctl status docker
```

**Expected Output:**
```
● docker.service - Docker Application Container Engine
   Loaded: loaded
   Active: active (running)
```

**If you see "active (running)"** → Docker is installed successfully!

### Common Installation Issues

#### Issue 1: Permission Denied

**Problem:**
```bash
docker run hello-world
# Error: permission denied
```

**Why:** Docker runs as root user by default

**Solution:** Add your user to docker group

```bash
sudo usermod -aG docker $USER
```

**Then:** Logout and login again

```bash
exit
# SSH back in
ssh -i your-key.pem ubuntu@your-ec2-ip
```

**Verify:**
```bash
docker run hello-world
```

Should work now!

#### Issue 2: Docker Daemon Not Running

**Problem:**
```bash
docker ps
# Error: Cannot connect to Docker daemon
```

**Solution:** Start Docker daemon

```bash
sudo systemctl start docker
sudo systemctl enable docker
```

#### Issue 3: User Not in Docker Group

**Check if user is in docker group:**
```bash
groups
```

**Should see:** `ubuntu docker`

**If not, add user:**
```bash
sudo usermod -aG docker ubuntu
```

**Then logout and login again!**



### Why Logout/Login is Needed

**Technical Reason:**
- User groups are loaded at login
- Adding to docker group doesn't update current session
- Need to reload user profile

**Two Ways to Fix:**
1. Logout and login (recommended)
2. Run: `newgrp docker` (temporary)

## Part 6: Your First Docker Project

### Project Overview

**What We'll Build:**
Simple Python application that prints "Hello World"

**Why This Example:**
- Very simple to understand
- Focuses on Docker concepts
- Easy to modify and experiment

**Files Needed:**
1. app.py (Python application)
2. Dockerfile (Build instructions)

### Step 1: Clone Repository

```bash
git clone https://github.com/[instructor-repo]/docker-zero-to-hero.git
cd docker-zero-to-hero/examples
```

**OR Create Files Manually:**

### Step 2: Create Application File

**Create app.py:**
```bash
nano app.py
```

**Add this code:**
```python
print("Hello World from Docker!")
```

**Save and exit:** Ctrl+X, then Y, then Enter

### Step 3: Create Dockerfile

**Create Dockerfile:**
```bash
nano Dockerfile
```

**Add these instructions:**
```dockerfile
FROM ubuntu:latest
WORKDIR /app
COPY app.py /app/
RUN apt-get update && apt-get install -y python3
CMD ["python3", "app.py"]
```

**Save and exit:** Ctrl+X, then Y, then Enter



### Understanding the Dockerfile

Let's break down each line:

#### Line 1: FROM ubuntu:latest
```dockerfile
FROM ubuntu:latest
```

**What it does:** Gets Ubuntu base image from Docker Hub

**Explanation:**
- `FROM` = Start with this base image
- `ubuntu:latest` = Ubuntu image, latest version
- Downloaded from Docker Hub

**Why needed:** Every container needs a base image

#### Line 2: WORKDIR /app
```dockerfile
WORKDIR /app
```

**What it does:** Sets working directory to /app

**Explanation:**
- Like running `cd /app`
- All commands run in this directory
- Creates /app if doesn't exist

**Optional:** Can skip this line

#### Line 3: COPY app.py /app/
```dockerfile
COPY app.py /app/
```

**What it does:** Copies app.py into container

**Explanation:**
- `COPY` = Copy files from host to container
- `app.py` = Source file (on your machine)
- `/app/` = Destination (in container)

**Important:** This puts your code in the container!

#### Line 4: RUN apt-get update && apt-get install -y python3
```dockerfile
RUN apt-get update && apt-get install -y python3
```

**What it does:** Installs Python3 in container

**Explanation:**
- `RUN` = Execute command during build
- `apt-get update` = Update package list
- `apt-get install -y python3` = Install Python3
- `-y` = Auto-confirm installation

**Why needed:** Our app needs Python to run!

#### Line 5: CMD ["python3", "app.py"]
```dockerfile
CMD ["python3", "app.py"]
```

**What it does:** Runs Python app when container starts

**Explanation:**
- `CMD` = Command to run when container starts
- `["python3", "app.py"]` = Run this command
- This is the entry point

**Result:** Container runs your Python app!

### Why This Dockerfile is Powerful

**Without Docker:**
```
1. Create VM
2. Install Ubuntu
3. Install Python3
4. Copy app.py
5. Run python3 app.py

Time: 30 minutes
Repeatable: No
```

**With Docker:**
```
1. Write Dockerfile (once)
2. docker build
3. docker run

Time: 2 minutes
Repeatable: Always!
```

## Part 7: Building Your First Docker Image

### Step 1: Verify Files

**Check you have both files:**
```bash
ls
```

**Should see:**
```
app.py
Dockerfile
```

### Step 2: Build the Image

**Command:**
```bash
docker build -t my-first-docker-image:latest .
```

**Let's break this down:**

- `docker build` = Build an image
- `-t` = Tag the image (give it a name)
- `my-first-docker-image` = Image name
- `:latest` = Version tag
- `.` = Look for Dockerfile in current directory

### Understanding Image Tags

**Format:** `name:tag`

**Examples:**
```
my-first-docker-image:latest
my-first-docker-image:v1
my-first-docker-image:1.0
ubuntu:20.04
python:3.9
```

**Why use tags:**
- Identify different versions
- Easy to remember
- Better than image IDs

**Without tag:**
- Image gets random ID
- Hard to remember
- Difficult to manage

### Build Process Explained

**What happens when you run docker build:**

```
Step 1: Docker reads Dockerfile
Step 2: Downloads ubuntu:latest from Docker Hub
Step 3: Creates /app directory
Step 4: Copies app.py into container
Step 5: Runs apt-get update
Step 6: Installs python3
Step 7: Sets CMD to run python3 app.py
Step 8: Creates final image
Step 9: Tags image as my-first-docker-image:latest
```

**Output you'll see:**
```
Sending build context to Docker daemon
Step 1/5 : FROM ubuntu:latest
latest: Pulling from library/ubuntu
...
Step 2/5 : WORKDIR /app
...
Step 3/5 : COPY app.py /app/
...
Step 4/5 : RUN apt-get update && apt-get install -y python3
...
Step 5/5 : CMD ["python3", "app.py"]
...
Successfully built abc123def456
Successfully tagged my-first-docker-image:latest
```

**Time:** 1-3 minutes (first time)



### Step 3: Verify Image is Created

**Command:**
```bash
docker images
```

**Output:**
```
REPOSITORY                TAG       IMAGE ID       CREATED         SIZE
my-first-docker-image     latest    abc123def456   2 minutes ago   150MB
ubuntu                    latest    xyz789abc123   2 weeks ago     77.8MB
```

**What you see:**
- **REPOSITORY:** Image name
- **TAG:** Version
- **IMAGE ID:** Unique identifier
- **CREATED:** When it was built
- **SIZE:** Image size

**Success!** Your first Docker image is ready!

## Part 8: Running Your First Container

### Step 1: Run the Container

**Command:**
```bash
docker run -it my-first-docker-image:latest
```

**Breaking it down:**
- `docker run` = Create and start container
- `-it` = Interactive terminal
- `my-first-docker-image:latest` = Image to use

**Expected Output:**
```
Hello World from Docker!
```

**Congratulations!** You just ran your first Docker container!

### Understanding docker run

**What happened:**
```
1. Docker found image: my-first-docker-image:latest
2. Created new container from image
3. Started the container
4. Ran CMD: python3 app.py
5. Printed: Hello World from Docker!
6. Container exited (app finished)
```

### The -it Flag Explained

**-it = -i + -t**

**-i (interactive):**
- Keep STDIN open
- Allow input to container

**-t (tty):**
- Allocate pseudo-terminal
- See output properly

**When to use:**
- Interactive applications
- Debugging
- Seeing output

**When not needed:**
- Background services
- Web servers
- Daemons



### Step 2: List Running Containers

**Command:**
```bash
docker ps
```

**Output:**
```
CONTAINER ID   IMAGE     COMMAND   CREATED   STATUS    PORTS     NAMES
```

**Empty?** That's normal! Our container already finished and exited.

### Step 3: List All Containers (Including Stopped)

**Command:**
```bash
docker ps -a
```

**Output:**
```
CONTAINER ID   IMAGE                          COMMAND                  CREATED         STATUS                     
abc123         my-first-docker-image:latest   "python3 app.py"         2 minutes ago   Exited (0) 2 minutes ago
```

**What you see:**
- Container was created
- It ran python3 app.py
- It exited successfully (status 0)

### Container Lifecycle

```
Created → Running → Stopped → Deleted
   ↑         ↓
   └─────────┘
   (can restart)
```

**Our container:**
```
Created → Ran for 1 second → Printed message → Exited
```

### Running Container in Background

**For long-running apps (web servers, etc.):**

```bash
docker run -d my-first-docker-image:latest
```

**-d flag:**
- Detached mode
- Runs in background
- Returns container ID

**Note:** Our simple app exits immediately, so -d doesn't help here. We'll use it for web apps later!

## Part 9: Pushing Image to Docker Hub

### Why Push to Docker Hub?

**Reasons:**
1. Share with team members
2. Deploy to production
3. Backup your images
4. Version control
5. Public portfolio

**Like:** Pushing code to GitHub

### Step 1: Create Docker Hub Account

**Go to:** hub.docker.com

**Click:** Sign Up

**Provide:**
- Username (remember this!)
- Email
- Password

**Verify email** and login



### Step 2: Login to Docker Hub from CLI

**Command:**
```bash
docker login
```

**Prompts:**
```
Username: your-dockerhub-username
Password: your-dockerhub-password
```

**Success message:**
```
Login Succeeded
```

**Important:** Remember your username!

### Step 3: Tag Image Properly

**Current image name:**
```
my-first-docker-image:latest
```

**Need to rename to:**
```
your-username/my-first-docker-image:latest
```

**Why?** Docker Hub requires username prefix!

**Command:**
```bash
docker tag my-first-docker-image:latest your-username/my-first-docker-image:latest
```

**Replace `your-username` with your actual Docker Hub username!**

**Example:**
```bash
docker tag my-first-docker-image:latest abhishekf5/my-first-docker-image:latest
```

**Verify:**
```bash
docker images
```

**Should see both:**
```
REPOSITORY                              TAG       IMAGE ID
my-first-docker-image                   latest    abc123
your-username/my-first-docker-image     latest    abc123
```

**Note:** Same IMAGE ID = Same image, just different names!

### Step 4: Push to Docker Hub

**Command:**
```bash
docker push your-username/my-first-docker-image:latest
```

**Example:**
```bash
docker push abhishekf5/my-first-docker-image:latest
```

**What happens:**
```
The push refers to repository [docker.io/your-username/my-first-docker-image]
Pushing layer 1/5
Pushing layer 2/5
Pushing layer 3/5
Pushing layer 4/5
Pushing layer 5/5
latest: digest: sha256:abc123... size: 1234
```

## Important Notes

### Docker Daemon Runs as Root

**Security Implication:**
- Docker daemon has root privileges
- Containers can potentially access host
- Use proper security practices

**Best Practices:**
- Don't run untrusted images
- Use official images when possible
- Scan images for vulnerabilities
- Keep Docker updated

### Single Point of Failure

**Docker Limitation:**
- If daemon crashes, all containers stop
- This is why Podman exists
- We'll learn Podman later

**For Now:**
- Docker is industry standard
- Master Docker first
- Learn alternatives later

## Interview Preparation

### Key Points to Remember

**Question:** "Why are containers lightweight?"

**Answer:** "Containers are lightweight because they don't contain a full operating system. They only have minimal system dependencies for logical isolation (like /bin, /etc, /lib directories). The kernel, file system, and networking stack are shared from the host OS, reducing size from gigabytes to megabytes."

**Question:** "What is Docker?"

**Answer:** "Docker is a containerization platform that provides tools to build, run, and share containers. It consists of Docker Client (CLI), Docker Daemon (engine), and integrates with Docker Hub for image sharing."

**Question:** "Explain Docker lifecycle."

**Answer:** "Docker lifecycle has three stages: First, you write a Dockerfile with build instructions. Second, you run 'docker build' to create an image. Third, you run 'docker run' to create and start a container from that image."

**Question:** "Difference between Docker Hub and GitHub?"

**Answer:** "GitHub stores source code and uses Git for version control. Docker Hub stores Docker images and uses tags for versioning. GitHub is for code collaboration, Docker Hub is for image sharing."
