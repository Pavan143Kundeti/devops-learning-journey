# Day 22 - Project Management for DevOps Engineers

## Introduction
This guide explains what a DevOps engineer does when they join an organization. It covers important project management tools and concepts beyond technical skills.

**Who needs this?**
- People switching careers to DevOps
- Fresh graduates joining as DevOps engineers
- People with career gaps
- Anyone wanting to understand DevOps project management

---

## What Happens When You Join an Organization?

### First Week:
- Understand the team structure
- Learn project management tools
- Get access to systems
- Understand workflows

### First Month:
- Start working on small tasks
- Track work using project tools
- Attend team meetings
- Learn company processes

**Important:** Technical skills alone are not enough. You need to know how to manage and track your work!

---

## Tool 1: Agile Methodology

### What is Agile?

**Simple Definition:**
Agile means working in small steps instead of doing everything at once.

### Traditional Way (Waterfall Method):

```
Step 1: Design entire application (2 months)
         ↓
Step 2: Write all code (3 months)
         ↓
Step 3: Test everything (2 months)
         ↓
Step 4: Give to customer (7 months total!)
```

**Problem:**
- Takes too long
- Customer waits 7 months
- If something is wrong, you wasted 7 months!

### Agile Way:

```
Week 1-2: Design small part → Build → Test → Show customer
Week 3-4: Design next part → Build → Test → Show customer
Week 5-6: Design next part → Build → Test → Show customer
```

**Benefits:**
- Customer sees progress every 2 weeks
- Problems found early
- Easy to make changes
- Less risk

### Real Example:

**Building a Shopping Website:**

**Waterfall Way:**
- Spend 6 months building entire website
- Show to customer after 6 months
- Customer says: "I don't like the design"
- Start again! (6 months wasted)

**Agile Way:**
- Week 1-2: Build login page → Show customer
- Week 3-4: Build product listing → Show customer
- Week 5-6: Build shopping cart → Show customer
- Customer gives feedback at each step
- Make changes immediately

### Important Agile Terms:

#### 1. Sprint
**What:** Fixed time period to complete work
**Example:** 2 weeks or 3 weeks
**What happens:** Team works on specific tasks during this time

#### 2. Sprint Planning
**What:** Meeting to decide what to do in next sprint
**Who:** Entire team
**When:** Start of each sprint

#### 3. Sprint Retrospective
**What:** Meeting to discuss what went well and what didn't
**Who:** Entire team
**When:** End of each sprint

#### 4. Backlog
**What:** List of all tasks waiting to be done
**Example:** 
- Fix login bug
- Add payment feature
- Improve website speed

#### 5. Scrum Master
**What:** Person who helps team follow Agile process
**Role:** Removes obstacles, organizes meetings

#### 6. Product Owner
**What:** Person who decides what features to build
**Role:** Represents customer needs

### Why DevOps Engineers Need to Know Agile:

- 90% of companies use Agile
- You'll attend Sprint meetings
- You'll get tasks assigned in Sprints
- You'll report progress in Agile format

### Where to Learn Agile:

1. Wikipedia - Search "Agile methodology"
2. YouTube - Search "Agile for beginners"
3. Practice - Join Agile meetings in your company

**Time needed:** 2-3 days to understand basics

## Tool 2: Jira

### What is Jira?

**Simple Definition:**
Jira is a tool to track your work and tasks.

**Made by:** Atlassian company

### Why Use Jira?

#### Problem Without Jira:
```
Manager: "What did you do last week?"
You: "Umm... I fixed some bugs... I think..."
Manager: "Which bugs? Show me proof."
You: "I don't remember..."
```

#### Solution With Jira:
```
Manager: "What did you do last week?"
You: Opens Jira → Shows completed tasks
Manager: Can see everything you did with dates and details
```

### What Can You Do in Jira?

#### 1. Track Tasks
- See what work is assigned to you
- Mark tasks as "To Do", "In Progress", "Done"
- Add comments on your progress

#### 2. Create Tickets
- Report bugs
- Request new features
- Ask questions

#### 3. Track Team Progress
- See what entire team is doing
- Check project status
- View deadlines

#### 4. Generate Reports
- How many tasks completed this month?
- Which tasks are delayed?
- Team performance metrics

### Jira Terminology:

#### 1. Epic
**What:** Big feature that takes months
**Example:** "Build payment system"

#### 2. Story
**What:** Smaller part of Epic
**Example:** "Add credit card payment option"

#### 3. Task
**What:** Specific work item
**Example:** "Write code for credit card validation"

#### 4. Bug
**What:** Something that's broken
**Example:** "Login button not working"

#### 5. Sprint
**What:** Time period (2-3 weeks)
**Example:** "Sprint 23: March 1-14"

### Jira Workflow Example:

```
1. Manager creates Story: "Deploy application to Kubernetes"
         ↓
2. Story assigned to you in Jira
         ↓
3. You move it to "In Progress"
         ↓
4. You add comments: "Created Dockerfile"
         ↓
5. You add comments: "Pushed to Docker Hub"
         ↓
6. You move to "Done"
         ↓
7. Manager can see your progress anytime
```

### Real Jira Use Case:

**Scenario:** CEO wants to know all Priority 1 issues

**Without Jira:**
- Email all managers
- Wait for replies
- Compile information manually
- Takes 2 days

**With Jira:**
- Write simple query: "Priority = P1"
- Get instant list of all P1 issues
- Takes 2 minutes!

### How to Practice Jira:

#### Step 1: Create Free Account
1. Go to: https://www.atlassian.com/software/jira
2. Click "Start free trial" (30 days)
3. No credit card needed!

#### Step 2: Setup
1. Sign up with Gmail
2. Create organization name (any name)
3. Select "Jira Software"
4. Click "Generate License"

#### Step 3: Download and Install
1. Download Jira application
2. Install on your computer
3. Apply the free license
4. Start using!

#### Step 4: Practice
- Create dummy projects
- Add tasks
- Move tasks between columns
- Add comments
- Generate reports

**Time to learn:** 1-2 days of practice

### Jira Board Example:

```
┌─────────────┬──────────────┬──────────────┬─────────┐
│   TO DO     │ IN PROGRESS  │   REVIEW     │  DONE   │
├─────────────┼──────────────┼──────────────┼─────────┤
```

You drag tasks from left to right as you complete them!

## Tool 3: Confluence

### What is Confluence?

**Simple Definition:**
Confluence is like Wikipedia for your company. It stores all company knowledge and documents.

**Made by:** Atlassian (same company as Jira)

### Why Use Confluence?

#### The Problem:
```
New employee joins: "How do I deploy to production?"
Senior employee: "Let me explain... it takes 2 hours"
Another new employee joins next month: Same 2-hour explanation!
Senior employee leaves: Knowledge is lost forever!
```

#### The Solution:
```
Senior employee writes document in Confluence once
New employees read document anytime
Knowledge stays in company forever
Everyone saves time!
```

### What to Store in Confluence:

#### 1. Technical Documentation
- How to deploy applications
- How to access servers
- How to use internal tools

#### 2. Process Documents
- How to request access
- How to report bugs
- How to escalate issues

#### 3. Architecture Diagrams
- System design
- Network diagrams
- Database schemas

#### 4. Meeting Notes
- Sprint planning notes
- Decision records
- Action items

#### 5. Runbooks
- What to do when server crashes
- How to handle incidents
- Emergency procedures

### Real Example:

**Scenario:** Senior DevOps engineer built entire CI/CD system

**Without Confluence:**
- Knowledge only in his head
- He leaves company
- New team struggles
- Takes months to understand

**With Confluence:**
- He documented everything
- Step-by-step guides
- Screenshots included
- New team reads and understands in days

### Confluence Page Structure:

```
Company Confluence
│
├── DevOps Team Space
│   ├── CI/CD Documentation
│   │   ├── Jenkins Setup Guide
│   │   ├── Pipeline Examples
│   │   └── Troubleshooting
│   │
│   ├── Kubernetes Documentation
│   │   ├── Cluster Setup
│   │   ├── Deployment Guide
│   │   └── Common Issues
│   │
│   └── Runbooks
│       ├── Production Deployment
│       ├── Incident Response
│       └── Backup Procedures
│
└── Development Team Space
    └── API Documentation
```

### How to Practice Confluence:

1. Use same Atlassian account from Jira
2. Access Confluence with same login
3. Create test pages
4. Add images, tables, code blocks
5. Practice organizing information

**Time to learn:** 1 day

## Tool 5: ServiceNow

### What is ServiceNow?

**Simple Definition:**
ServiceNow is a tool to manage incidents and changes in production systems.

### Two Main Uses:

#### 1. Incident Management
#### 2. Change Management

Let's understand both:

---

### Part 1: Incident Management

#### What is an Incident?

**Simple Definition:**
Something broke in production and needs immediate fix!

#### Real Example:

**Scenario:**
```
Time: 2:00 AM
Event: Website crashes
Impact: Customers can't buy products
Status: INCIDENT!
```

#### How Incident Management Works:

**Step 1: Detection**
```
Monitoring system (like Prometheus) detects problem
         ↓
Alert sent: "Website is down!"
```

**Step 2: Create Incident**
```
ServiceNow automatically creates incident ticket
Incident ID: INC0012345
Priority: P1 (Critical)
Description: "Website not responding"
```

**Step 3: Assignment**
```
ServiceNow assigns to on-call engineer
Engineer gets notification on phone
Engineer starts working immediately
```

**Step 4: Resolution**
```
Engineer fixes the problem
Engineer adds comments in ServiceNow:
  - "Found memory leak in payment service"
  - "Restarted service"
  - "Website is back online"
```

**Step 5: Closure**
```
Engineer marks incident as "Resolved"
Manager reviews and closes ticket
Post-mortem meeting scheduled
```

#### Incident Priority Levels:

**P1 - Critical:**
- Production completely down
- All customers affected
- Fix immediately (within 1 hour)

**P2 - High:**
- Major feature not working
- Many customers affected
- Fix within 4 hours

**P3 - Medium:**
- Minor feature issue
- Some customers affected
- Fix within 24 hours

**P4 - Low:**
- Small bug
- Few customers affected
- Fix within 1 week

#### ServiceNow Incident Workflow:

```
┌──────────────────────────────────────────────────┐
│  Monitoring System Detects Problem               │
└────────────────┬─────────────────────────────────┘
                 ↓
┌──────────────────────────────────────────────────┐
│  ServiceNow Creates Incident Automatically       │
│  - Incident Number: INC0012345                   │
│  - Priority: P1                                  │
│  - Assigned to: DevOps Team                      │
└────────────────┬─────────────────────────────────┘
                 ↓
┌──────────────────────────────────────────────────┐
│  Engineer Receives Alert                         │
│  - Email notification                            │
│  - SMS notification                              │
│  - Phone call (for P1)                           │
└────────────────┬─────────────────────────────────┘
                 ↓
┌──────────────────────────────────────────────────┐
│  Engineer Investigates                           │
│  - Checks logs                                   │
│  - Identifies root cause                         │
│  - Updates ServiceNow with findings              │
└────────────────┬─────────────────────────────────┘
                 ↓
┌──────────────────────────────────────────────────┐
│  Engineer Fixes Problem                          │
│  - Applies fix                                   │
│  - Tests solution                                │
│  - Updates ServiceNow                            │
└────────────────┬─────────────────────────────────┘
                 ↓
┌──────────────────────────────────────────────────┐
│  Incident Resolved                               │
│  - Mark as resolved in ServiceNow                │
│  - Manager reviews                               │
│  - Incident closed                               │
└──────────────────────────────────────────────────┘
```

### Part 2: Change Management

#### What is Change Management?

**Simple Definition:**
A process to safely make changes to production systems without breaking anything.

#### Why Do We Need It?

**Without Change Management:**
```
Engineer: "I'll quickly fix this bug in production"
         ↓
Makes change without telling anyone
         ↓
Entire system crashes
         ↓
Nobody knows what happened
         ↓
Takes hours to fix
```

**With Change Management:**
```
Engineer: "I need to fix a bug"
         ↓
Creates change request in ServiceNow
         ↓
Team reviews the change
         ↓
Change approved with proper planning
         ↓
Change made safely during maintenance window
         ↓
Everything works smoothly
```

#### Change Management Process:

**Step 1: Create Change Request**
```
Engineer creates ticket in ServiceNow:
- What needs to change?
- Why is it needed?
- When to do it?
- What's the risk?
- What's the rollback plan?
```

**Step 2: Review and Approval**
```
Change Advisory Board (CAB) reviews:
- Technical team checks feasibility
- Manager checks business impact
- Security team checks security
- All approve or reject
```

**Step 3: Schedule Change**
```
Change scheduled for:
- Low traffic time (e.g., 2 AM Sunday)
- Maintenance window
- When support team is available
```

**Step 4: Execute Change**
```
During scheduled time:
- Notify all teams
- Take backup
- Make the change
- Test thoroughly
- Monitor closely
```

**Step 5: Verify and Close**
```
After change:
- Verify everything works
- Update ServiceNow
- Close change ticket
- Document lessons learned
```

#### Types of Changes:

**1. Standard Change:**
- Low risk
- Done many times before
- Pre-approved
- Example: Restarting a service

**2. Normal Change:**
- Medium risk
- Needs approval
- Proper planning required
- Example: Deploying new feature

**3. Emergency Change:**
- High risk but urgent
- Fast-track approval
- Production is down
- Example: Fixing critical security bug

#### Real Example:

**Scenario:** Memory leak found in production payment service

**Change Request Details:**
```
Change ID: CHG0045678
Title: Fix memory leak in payment service
Priority: High
Risk: Medium
Scheduled: Sunday 2:00 AM - 4:00 AM

Description:
- Payment service has memory leak
- Causes crash every 6 hours
- Need to deploy fixed version

Impact:
- Payment service down for 15 minutes
- Customers can't make payments during this time

Teams Involved:
- DevOps: Deploy new version
- Database: Backup payment data
- Network: Monitor traffic
- Support: Handle customer queries

Rollback Plan:
- If new version fails, rollback to previous version
- Rollback takes 5 minutes
- Database backup available

Approval:
✓ Technical Lead: Approved
✓ Manager: Approved
✓ Security: Approved
```

#### Change Management Best Practices:

1. **Always have rollback plan**
   - What if change fails?
   - How to go back to previous state?

2. **Test in non-production first**
   - Test in dev environment
   - Test in staging environment
   - Then deploy to production

3. **Schedule during low traffic**
   - Weekends
   - Late night
   - Maintenance windows

4. **Communicate clearly**
   - Notify all teams
   - Update customers if needed
   - Keep everyone informed

5. **Document everything**
   - What was changed
   - Why it was changed
   - What was the result

## Tool 6: Read the Docs

### What is Read the Docs?

**Simple Definition:**
Read the Docs is a free platform to host documentation websites for open-source projects.

**Website:** readthedocs.org

### Why Use Read the Docs?

#### For Open Source Projects:

**Problem:**
- Confluence and SharePoint cost money
- Open source projects need free tools
- Need to share documentation publicly

**Solution:**
- Read the Docs is free
- Easy to use
- Integrates with GitHub
- Automatic updates

### How It Works:

```
1. Write documentation in Markdown files
         ↓
2. Store in GitHub repository
         ↓
3. Connect GitHub to Read the Docs
         ↓
4. Read the Docs builds website automatically
         ↓
5. Documentation available at: yourproject.readthedocs.io
```

### Real Example:

**Popular projects using Read the Docs:**
- Python documentation
- Django documentation
- Flask documentation
- Many others

### Features:

1. **Automatic Building**
   - Push to GitHub
   - Website updates automatically

2. **Version Control**
   - Different versions of docs
   - Users can switch versions

3. **Search Functionality**
   - Built-in search
   - Find information quickly

4. **Free Hosting**
   - No cost for open source
   - Fast and reliable

### Read the Docs vs Confluence:

| Feature | Read the Docs | Confluence |
|---------|---------------|------------|
| Cost | Free | Paid |
| Best for | Open source | Companies |
| Access | Public | Private |
| Integration | GitHub | Jira |
| Hosting | Cloud | Cloud/On-premise |
