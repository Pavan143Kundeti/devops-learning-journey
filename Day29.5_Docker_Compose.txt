# Docker Compose - Simple Guide

## WHAT IS DOCKER COMPOSE?

**Simple Answer:** Tool to run MULTIPLE containers together easily.

---

## THE PROBLEM

### With Docker (Single Container):

You learned Docker. It works great for ONE container!

```bash
docker build -t myapp .
docker run myapp
```

Easy!

### But What About Multiple Containers?

Real applications have MANY containers:

```
E-commerce App:
├── Frontend container
├── Backend container
├── Database container
├── Redis cache container
├── Payment container
└── Shipping container
```

**Problem:**
- Must run `docker build` 6 times
- Must run `docker run` 6 times
- Must remember which container starts first
- Database must start BEFORE backend
- Backend must start BEFORE frontend
- Too many commands!
- Developer gets confused!

---

## THE SOLUTION: DOCKER COMPOSE

**One command to start ALL containers:**

```bash
docker-compose up
```

**One command to stop ALL containers:**

```bash
docker-compose down
```

That's it! 🎉

---

## REAL-WORLD EXAMPLE

### Without Docker Compose (Hard Way):

```bash
# Start database
docker build -t mydb ./database
docker run -d --name db mydb

# Start backend (must wait for database!)
docker build -t mybackend ./backend
docker run -d --name backend --link db mybackend

# Start frontend (must wait for backend!)
docker build -t myfrontend ./frontend
docker run -d --name frontend --link backend myfrontend

# Start redis
docker build -t myredis ./redis
docker run -d --name redis myredis
```

Too many commands! Easy to make mistakes!

### With Docker Compose (Easy Way):

```bash
docker-compose up
```

Done! All containers start in correct order!

---

## HOW DOCKER COMPOSE WORKS

### Step 1: Write ONE File (docker-compose.yml)

This file tells Docker Compose:
- What containers to run
- In what order
- How they connect

### Step 2: Run ONE Command

```bash
docker-compose up
```

Docker Compose reads the file and starts everything!

---

## SIMPLE EXAMPLE

### Project Structure:

```
myapp/
├── frontend/
│   └── Dockerfile
├── backend/
│   └── Dockerfile
├── database/
│   └── Dockerfile
└── docker-compose.yml  ← Magic file!
```

### docker-compose.yml File:

```yaml
services:
  database:
    build: ./database
    
  backend:
    build: ./backend
    depends_on:
      - database
    
  frontend:
    build: ./frontend
    depends_on:
      - backend
    ports:
      - "80:80"
```

**What this means:**
- `services:` = List of containers
- `database:` = First container name
- `build: ./database` = Where Dockerfile is
- `depends_on:` = Start database FIRST, then backend
- `ports:` = Map port 80

## PRACTICAL EXAMPLE: WEB APP

### Scenario: Website with Database

**What we need:**
- Nginx (web server)
- Node.js (backend)
- Redis (cache)

### docker-compose.yml:

```yaml
services:
  redis:
    image: redis
    ports:
      - "6379:6379"
  
  backend:
    build: ./backend
    ports:
      - "5000:5000"
    depends_on:
      - redis
  
  nginx:
    build: ./nginx
    ports:
      - "80:80"
    depends_on:
      - backend
```

### Run It:

```bash
# Start everything
docker-compose up

# Stop everything
docker-compose down
```

## DOCKER COMPOSE COMMANDS

```bash
# Start all containers
docker-compose up

# Start in background
docker-compose up -d

# Stop all containers
docker-compose down

# See running containers
docker-compose ps

# See logs
docker-compose logs

# Restart all
docker-compose restart

# Build images
docker-compose build
```

## KEY FEATURES

### 1. Dependencies (Order Matters!)

```yaml
services:
  database:
    image: postgres
  
  backend:
    build: ./backend
    depends_on:
      - database  # Start database FIRST!
```

Database starts first, then backend!

### 2. Port Mapping

```yaml
services:
  web:
    build: ./web
    ports:
      - "8080:80"  # Host:Container
```

Access on localhost:8080

### 3. Environment Variables

```yaml
services:
  database:
    image: postgres
    environment:
      POSTGRES_PASSWORD: secret123
```

### 4. Volumes (Save Data)

```yaml
services:
  database:
    image: postgres
    volumes:
      - db-data:/var/lib/postgresql/data

volumes:
  db-data:
```

## COMPLETE EXAMPLE

### Project: Blog Website

**Structure:**
```
blog/
├── frontend/
│   └── Dockerfile
├── backend/
│   └── Dockerfile
└── docker-compose.yml
```

### docker-compose.yml:

```yaml
services:
  # Database
  database:
    image: postgres:13
    environment:
      POSTGRES_PASSWORD: secret
    volumes:
      - db-data:/var/lib/postgresql/data
  
  # Backend API
  backend:
    build: ./backend
    ports:
      - "5000:5000"
    depends_on:
      - database
    environment:
      DATABASE_URL: postgres://database:5432
  
  # Frontend
  frontend:
    build: ./frontend
    ports:
      - "80:80"
    depends_on:
      - backend

volumes:
  db-data:
```

### Run:

```bash
# Start
docker-compose up

# Visit: http://localhost

# Stop
docker-compose down
```

## WHEN TO USE DOCKER COMPOSE?

### ✅ Use Docker Compose For:

1. **Local Development**
   - Developer testing on laptop
   - Quick start/stop
   - Easy setup

2. **Testing**
   - QA team testing
   - CI/CD pipelines
   - Quick verification

3. **Small Projects**
   - Personal projects
   - Demos
   - Prototypes

### ❌ Don't Use Docker Compose For:

1. **Production (Large Scale)**
   - Use Kubernetes instead
   - Need auto-scaling
   - Need auto-healing
   - Multiple servers

2. **High Availability**
   - Docker Compose = Single machine
   - Kubernetes = Multiple machines

---

## DOCKER COMPOSE vs KUBERNETES

| Feature | Docker Compose | Kubernetes |
|---------|----------------|------------|
| Machines | One computer | Many computers |
| Auto-healing | ❌ No | ✅ Yes |
| Auto-scaling | ❌ No | ✅ Yes |
| Load balancing | Basic | Advanced |
| Good for | Development | Production |
| Complexity | Easy | Complex |

**Simple Rule:**
- Development/Testing → Docker Compose
- Production → Kubernetes

## COMMON MISTAKES

### Mistake 1: Wrong Order

```yaml
# ❌ BAD
services:
  backend:
    build: ./backend
  
  database:
    image: postgres
```

Backend starts before database = ERROR!

```yaml
# ✅ GOOD
services:
  database:
    image: postgres
  
  backend:
    build: ./backend
    depends_on:
      - database
```

### Mistake 2: Forgetting Ports

```yaml
# ❌ BAD - Can't access!
services:
  web:
    build: ./web
```

```yaml
# ✅ GOOD
services:
  web:
    build: ./web
    ports:
      - "80:80"
```

## IMPORTANT NOTES

### Docker Compose Needs Docker!

- Install Docker first
- Docker Compose comes with Docker Desktop
- On Linux: Install separately

### One docker-compose.yml Per Project

```
project1/
└── docker-compose.yml

project2/
└── docker-compose.yml
```

### Dockerfiles Still Needed!

Docker Compose doesn't replace Dockerfiles!

```
You still need:
- Dockerfile (how to build container)
- docker-compose.yml (how to run multiple containers)
```

## TROUBLESHOOTING

### Problem: Containers won't start

```bash
# Check logs
docker-compose logs

# Check specific service
docker-compose logs backend
```

### Problem: Port already in use

```bash
# Change port in docker-compose.yml
ports:
  - "8080:80"  # Use 8080 instead of 80
```

### Problem: Changes not appearing

```bash
# Rebuild images
docker-compose build

# Restart
docker-compose up
```

## KEY TAKEAWAYS

1. **Docker Compose = Multiple containers, one command**
2. **One file (docker-compose.yml) controls everything**
3. **Perfect for development and testing**
4. **Use Kubernetes for production**
5. **Commands:**
   - `docker-compose up` = Start
   - `docker-compose down` = Stop
6. **Dependencies matter** (use `depends_on`)
7. **Still need Dockerfiles!**
